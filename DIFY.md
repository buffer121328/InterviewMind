# Dify DSL 版本说明

仓库根目录提供了一个 Dify 应用 DSL：`面必过chatbot.yml`。它是 lite 技能版的轻量延伸：保留「简历优化 + 模拟面试」这条纯文本交互流程，但用 Dify 的可视化管线、会话变量和结构化输出把关键边界固定住。

这个版本不关注后端服务、数据库、代码部署或传统 Web 应用改造。它的目标是让非编程用户也能通过一个在线聊天入口完成文件上传、JD 输入、简历优化和模拟面试复盘。

## 定位

| 版本 | 适合场景 | 核心定位 |
|------|----------|----------|
| Codex Skills 版 | 个人快速试用、本地材料处理、语音面试练习 | 零门槛、灵活、Agent 本地执行 |
| Dify DSL 版 | 在线入口、稳定流程、结构化结果、非编程用户使用 | 可视化编排、schema 约束、轻量交付 |

简单说：Skills 负责把门槛降到最低，让用户愿意先试；Dify 负责把流程固定住，让结果更稳定、更适合交付给不熟悉 Agent 和命令行的人。

## 如何导入

1. 打开 Dify 控制台，进入应用创建/导入入口。
2. 选择从 DSL/YAML 导入，上传本仓库的 `面必过chatbot.yml`。
3. 按 Dify 提示安装或授权 DSL 中声明的依赖：
   - `langgenius/deepseek`：用于 `deepseek-v4-flash` 和 `deepseek-v4-pro` 模型节点。
   - `bowenliang123/md_exporter`：用于把 Markdown 报告转换为 HTML 文件。
4. 在 Dify 中配置自己的模型供应商凭据。不要把 API Key 写进仓库或提交到 Git。
5. 发布/运行应用后，在聊天窗口上传简历文档，并在「职位介绍」里填写目标岗位 JD。

## 如何使用

### 简历优化

上传简历，填写 JD，然后发送：

```
优化简历
```

Dify 工作流会自动进入简历链路：

1. 用文档提取器读取简历内容。
2. 结构化整理简历信息，不虚构经历、技术栈和数据。
3. 由 JD 匹配分析师评估技能、项目、经验、教育四个维度。
4. 由内容优化师给出 STAR 改写、关键词补强和删减建议。
5. 由 HR 审核节点模拟 6 秒初筛，输出亮点、顾虑、通过率预估和优先级。
6. 由圆桌式复盘节点整合结果，生成优化后 Markdown 简历。
7. 生成适合导出的 HTML 简历文件。

### 模拟面试

上传简历，填写 JD，然后发送：

```
模拟面试
```

Dify 工作流会自动进入面试链路：

1. 先提取并保存结构化简历信息和 JD。
2. 面试引擎基于简历、JD 和历史问答逐轮提问。
3. 每轮只问一个主问题，内部按开场、技术深挖、项目实战、行为问题、反问、总结推进。
4. 面试题提取器保存当前问题，问答收集器把你的回答追加到 `qa_history`。
5. 当面试结束后，表现评估专家按 6 个维度给出 0-100 分评估。
6. 改进方案专家输出知识盲区、最差回答、行动计划和推荐练习题。
7. 生成模拟面试综合复盘报告，并导出 HTML 文件。

## 这一版 Dify DSL 的功能

| 能力 | 说明 |
|------|------|
| 意图识别 | 自动识别「模拟面试」「简历优化」和其他闲聊，并分流到对应链路 |
| 状态锁 | 用 `current_mode` 保持当前流程，避免多轮对话时频繁重新判断意图 |
| 简历解析 | 使用 Dify 文档提取器读取上传的简历文档 |
| JD 匹配 | 输出关键词、缺失关键词、总匹配分和四维匹配分析 |
| 多专家简历优化 | JD 匹配分析师、内容优化师、HR 审核、圆桌整合多节点串联 |
| 结构化输出 | 多个 LLM 节点启用 structured output，方便下游节点稳定读取字段 |
| 6 阶段模拟面试 | 面试官按真实面试节奏逐轮追问，不直接暴露阶段和评分 |
| 问答历史 | 用会话变量保存上一题、历史问答、简历摘要和 JD |
| 面试复盘 | 结束后生成 6 维表现评估、知识盲区、行动计划和推荐练习题 |
| HTML 导出 | 通过 `md_exporter` 把简历和复盘报告导出为 HTML 文件 |

## 局限与风险

Dify 版本把流程固定得更清楚，但它不是无成本替代，主要问题如下：

| 问题 | 说明 |
|------|------|
| 回复耗时更长 | 简历优化和面试复盘会串联多个 LLM 节点、代码节点和 HTML 导出工具，整体比 Skills 版直接对话慢很多 |
| 依赖提示词规范性 | schema 能约束字段形状，但每个专家节点的判断质量仍然依赖提示词；提示词写得不清楚时，结果会变空泛、重复或跑偏 |
| 线上版本容易出问题 | Dify 云端、模型供应商、插件服务、文件解析和网络任一环节波动，都可能导致节点失败、超时或输出缺失 |
| 调试链路更长 | 一个结果异常可能来自意图识别、文档提取、变量传递、结构化输出、代码清洗或导出工具，定位成本比单个 Agent 对话更高 |
| 流程灵活度较低 | 工作流是流水线式编排，适合固定场景；用户临时改变目标、补充材料或跳出流程时，需要额外设计分支 |
| 文件解析有边界 | 扫描版 PDF、排版复杂的简历、图片化内容和格式异常的文档，可能提取不完整，后续评分和优化会被连带影响 |
| 结构化输出不等于正确 | structured output 只能保证字段更稳定，不能保证评分、公平性、改写质量和面试判断一定准确 |
| 维护成本偏高 | 节点多以后，schema、变量名、模型版本、插件版本和提示词之间会互相影响，小改动也需要端到端重测 |
| 隐私依赖平台 | 在线入口更方便非编程用户，但简历和 JD 会经过 Dify 与模型供应商链路，需要按实际部署方式评估隐私和合规 |

所以这个 DSL 更适合作为 lite 技能版之后的稳定在线入口，而不是替代所有本地 Agent 能力。它换来的是可视化编排和结构化结果，付出的代价是速度、平台依赖和维护复杂度。

## Skills 版 vs Dify DSL 版

| 对比项 | Codex Skills 版 | Dify DSL 版 |
|---|---|---|
| 运行方式 | 在本地项目目录中由 codex 读取技能并执行 | 在 Dify advanced-chat 工作流中运行 |
| 上手成本 | 适合已经在用 codex 的个人用户，拷贝 `skills/` 即可 | 用户侧只需要打开在线入口；维护者侧需要导入 DSL、配置模型和插件 |
| 编排方式 | 通过技能说明和 Agent 自主规划完成 | 可视化节点编排，流程结构直观，方便审查每个节点 |
| 工具使用 | Agent 按技能说明和上下文选择工具，可访问本地文件和浏览器自动化 | 工具节点由工作流显式指定，例如文档提取器、Markdown 转 HTML |
| 输出稳定性 | 灵活，但强依赖 Agent 对技能说明的理解 | structured output 更适合把字段传给后续节点 |
| 可扩展性 | 容易接入本地脚本、文件、浏览器、命令行和项目资料 | 容易接入 Dify 插件、在线聊天入口、团队共享和后续飞书等消息界面 |
| 速度 | 通常链路更短，交互更轻 | 多节点流水线串行执行，回复会明显更慢 |
| 灵活度 | Agent 可根据材料临场调整流程、追问和产物形式 | 流水线式使用，流程固定，临场变化需要改工作流 |
| 隐私与依赖 | 主要依赖 codex 当前环境，不额外要求 Dify 服务 | 依赖 Dify 服务、模型供应商、插件授权和网络环境 |
| 维护成本 | 维护 Markdown 技能文件和少量脚本 | 维护 DSL 节点、变量、结构化 schema、插件版本和模型配置 |
| 主要优点 | 本地化、灵活、能直接处理文件和自动化任务 | 可视化编排、结构化输出、工具使用可控、适合非编程用户在线使用 |
| 主要缺点 | 对 Agent 能力和上下文管理要求更高，团队协作可视化弱 | 耗时长、流水线较重、依赖提示词质量，线上服务和插件链路容易带来不稳定，节点多时调试和版本迁移成本高 |

简单说：**个人求职、想快速用起来，优先用 Codex Skills 版；想提供一个不用装环境、不用看代码的稳定在线入口，就用 Dify DSL 版。**
