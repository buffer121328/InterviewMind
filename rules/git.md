# Git 规范

## 忽略文件追踪

- 严格遵守 `.gitignore`：被忽略的文件或目录默认不得通过修改 `.gitignore`、`git add -f` 或其他方式强行纳入追踪。
- 不要为了让某个新文件出现在 `git status` 中而擅自添加 `.gitignore` 例外规则。
- 只有在用户明确要求追踪某个被忽略路径时，才可以修改 `.gitignore` 或使用强制添加，并必须说明影响范围。
- `.env`、`.env.*`、浏览器 profile、数据库备份、日志、临时输出和真实密钥文件不得提交。

## 暂存与提交

- 提交前只暂存属于当前任务范围、且未被 `.gitignore` 忽略的文件。
- 不要把用户或其他工具产生的无关改动混入同一次提交。
- 阶段完成后，先确认验收标准、相关测试和 README 已同步，再进行暂存与提交。
- 阶段提交信息应说明 phase/功能范围、主要验收点和是否包含安全/配置变更。
- 如用户要求提交，提交前先展示将暂存的文件清单；涉及迁移、配置、安全策略时说明影响范围。

## 分支与历史

- 新分支默认使用 `codex/` 前缀，除非用户指定其他命名。
- 不要未经用户明确要求执行破坏性历史操作，例如 `git reset --hard`、强推、rebase 公共分支或清理未跟踪文件。
- 修复问题时优先小提交、单一主题提交；避免大范围格式化造成审查噪音。
