# 评测模块 — 架构设计与技术方案

## 1. 背景与目标

### 1.1 现状

项目当前**零评测覆盖**。仅有 3 个单元测试文件（RAG、JD matcher、candidate materials），对以下核心能力完全没有评测：

- LLM 生成的面试问题质量
- 面试对话的追问逻辑正确性
- 候选人评分的准确性
- 简历优化的忠实度和有效性
- LangGraph 状态机的流转正确性
- Prompt/模型变更后的质量回归

### 1.2 目标

建立一套**可持续运行的评测体系**，实现：

1. **质量可量化** — 每个核心环节都有可测量的指标
2. **变更可验证** — prompt/模型变更后，自动检测质量是否退化
3. **问题可定位** — 指标下降时能快速定位到具体环节
4. **成本可控** — 日常开发跑快速回归，完整评测按需运行

## 2. 技术选型

### 2.1 方案对比

| 方案 | 契合度 | 成本 | 灵活性 | 维护负担 | 结论 |
|------|--------|------|--------|----------|------|
| **DeepEval** | ⭐⭐⭐⭐ 高 | 开源免费 | GEval 支持自定义维度 | 低（pytest 原生集成） | ✅ 采用 |
| Ragas | ⭐⭐ 低 | 开源免费 | 指标体系为 RAG 设计 | 中 | ❌ 不匹配 |
| LangSmith | ⭐⭐⭐ 中高 | 免费额度有限 | 厂商锁定 | 低 | ❌ 成本+锁定 |
| Promptfoo | ⭐⭐⭐ 中 | 开源免费 | CLI 工具，集成成本高 | 中 | ❌ 集成不便 |
| OpenAI Evals | ⭐ 低 | 开源免费 | 强绑定 OpenAI | 高 | ❌ 不兼容 |
| 纯手搓 | ⭐⭐⭐⭐⭐ | 0 | 完全可控 | 极高 | ❌ ROI 太低 |

### 2.2 最终选型：DeepEval + 自定义评测器

**DeepEval 负责基础设施**：
- pytest 原生集成（`@pytest.mark.parametrize` + `assert_test`）
- LLM-as-judge 引擎（GEval 支持自然语言定义评分标准）
- 内置指标（HallucinationMetric、FaithfulnessMetric、AnswerRelevancyMetric）
- 评测结果报告

**自定义评测器负责领域逻辑**：
- 面试问题质量（JD 相关性、类型多样性、去重有效性）
- 追问逻辑正确性（_classify_responder_action 的边界测试）
- 状态机流转正确性（路由函数、节点执行序列）
- 评分校准度（需要标注数据集）
- 简历优化的 JD 关键词覆盖

### 3.3 测试分层策略

| 层级 | 运行时机 | LLM 消耗 | 耗时 | 覆盖范围 |
|------|---------|----------|------|---------|
| **L1: 工作流正确性** | 每次 PR | 0 tokens | <10s | 路由、分类、状态断言 |
| **L2: 回归测试** | 每次 PR | ~5k tokens | <2min | JSON 解析率、输出数量、兜底逻辑 |
| **L3: 质量评测** | main 合并 / 手动 | ~50k tokens | ~10min | JD 相关性、清晰度、幻觉检测 |
| **L4: 完整评测** | 发版前 / 周度 | ~200k tokens | ~30min | 全量 Golden Dataset + 标注数据校准 |

## 4. Mock 策略

### 4.1 为什么需要 Mock

项目的核心特点是 **API 配置由用户前端传入**（`api_config`），不是服务端固定配置。评测时需要：

1. **工作流测试**：完全 mock LLM，只验证逻辑流转
2. **质量评测**：使用固定的测试 API 配置，确保可复现

### 4.2 Mock 实现

```python
# tests/eval/conftest.py
```

## 5. 运行方式

### 5.1 命令行运行

```bash
# 运行全部评测
cd backend
source .venv/bin/activate
python -m pytest tests/eval/ -v

# 只运行工作流正确性（快速，0 成本）
python -m pytest tests/eval/test_workflow_correctness.py -v

# 只运行回归测试
python -m pytest tests/eval/test_regression.py -v

# 运行 LLM 质量评测（需要真实 API Key）
python -m pytest tests/eval/test_interview_planner.py -v

# 按标记运行
python -m pytest tests/eval/ -m "fast" -v      # 快速测试
python -m pytest tests/eval/ -m "llm" -v       # 需要 LLM 的测试
python -m pytest tests/eval/ -m "regression" -v # 回归测试
```

### 5.2 pytest 配置

```ini
# backend/pytest.ini
[pytest]
testpaths = tests
markers =
    fast: 快速测试（不消耗 LLM tokens）
    llm: 需要 LLM API 的测试
    regression: 回归测试套件
    eval: 质量评测（消耗较多 tokens）
asyncio_mode = auto
```

## 6. 成本估算

| 测试层级 | 每次运行 tokens | 单价（GPT-4o-mini） | 每次成本 |
|---------|----------------|---------------------|---------|
| L1 工作流正确性 | 0 | - | $0 |
| L2 回归测试 | ~5k | $0.15/1M input | <$0.01 |
| L3 质量评测 | ~50k | $0.15/1M input | ~$0.01 |
| L4 完整评测 | ~200k | $0.15/1M input | ~$0.03 |

> 使用 GPT-4o-mini 作为 judge 模型，成本极低。如果使用更强的模型（如 GPT-4o），成本约增加 10-20 倍。

## 7. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Judge LLM 偏差 | 评分不准确 | 用与被测不同的模型做 judge；关键指标多 judge 取平均 |
| LLM 输出不可复现 | 测试结果波动 | 回归测试用 `temperature=0`；质量评测设合理容差（threshold=0.7） |
| Golden Dataset 过时 | 评测失去意义 | 纳入版本管理；prompt 重大变更时同步更新 |
| 用户 API 配置隔离 | 测试无法运行 | conftest.py 提供固定测试配置；mock `get_llm_for_request` |
| Token 成本失控 | CI 费用飙升 | PR 只跑 L1+L2；L3/L4 仅在 main 分支或手动触发 |
| LangGraph 状态构造复杂 | 测试编写困难 | 提供 `build_test_state()` 工厂函数；节点函数设计为输入 state 输出 dict，可独立测试 |

## 8. 与现有代码的集成点

### 8.1 面试 Graph（interview_graph.py）

- `route_entry()` — 可直接单元测试，输入 state 返回路由目标
- `route_after_responder()` — 可直接单元测试
- `_classify_responder_action()` — 可直接单元测试，纯函数
- `node_planner()` — 需要 mock `interview_planner.generate_interview_plan`
- `node_responder()` — 需要 mock `create_react_agent`
- `node_summary()` — 需要 mock `interview_analysis.process_interview_summary`

### 8.2 面试规划（interview_planner.py）

- `build_planner_prompt()` — 纯函数，可直接测试 prompt 结构
- `parse_plan_response()` — 纯函数，可直接测试 JSON 解析
- `generate_interview_plan()` — 需要 mock `invoke_structured` 或使用真实 API

### 8.3 简历优化（resume_optimizer_graph.py）

- 各节点函数（`node_match_analyst` 等）— 需要 mock `invoke_structured`
- `build_resume_optimizer_graph()` — 可测试图结构
- `optimize_resume()` — 端到端测试，需要 mock LLM 或使用真实 API

### 8.4 LLM 调用（llm_utils.py）

- `invoke_structured()` — 核心调用链，质量评测的切入点
- `clean_json_response()` — 纯函数，可直接测试
