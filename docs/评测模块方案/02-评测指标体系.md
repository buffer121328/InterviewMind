# 评测模块 — 评测指标体系

## 概述

本项目评测体系覆盖 6 个维度，共 25+ 个具体指标。每个指标明确量化方式、评测方法和通过阈值。

---

## 维度 1：面试问题质量（Interview Question Quality）

**评测对象**：`interview_planner.generate_interview_plan()` 的输出

**输入**：简历 + JD + 轮次配置
**输出**：面试问题列表（`List[Dict]`，包含 id/topic/content/type/sources/reason）

### 指标列表

| # | 指标 | 量化方式 | 评测方法 | 阈值 | 优先级 |
|---|------|---------|---------|------|--------|
| 1.1 | **JD 相关性** | 问题内容与 JD 关键词的语义相似度 | DeepEval `GEval`：「评估面试问题是否与目标职位描述高度相关」 | ≥0.7 | P0 |
| 1.2 | **简历针对性** | `sources` 字段是否非空 + 证据相关性 | 自定义：检查每道题的 `sources` 列表，非空比例 | ≥60% 题目有 sources | P0 |
| 1.3 | **输出数量一致性** | 生成问题数 == 请求的 `max_questions` | 断言检查 | 100% | P0 |
| 1.4 | **类型分布合理性** | intro/tech/behavior/system_design 的比例是否符合 ROUND_STRATEGIES | 统计各类型占比，对比策略要求 | 偏离 ≤1 题 | P1 |
| 1.5 | **问题清晰度** | 问题是否具体、无歧义、可回答 | DeepEval `GEval`：「评估面试问题是否清晰具体」 | ≥0.8 | P1 |
| 1.6 | **去重有效性** | 多轮面试问题与 `previous_questions` 的重复率 | Jaccard 相似度（分词后） | ≤0.2 | P1 |
| 1.7 | **JSON 解析成功率** | `parse_plan_response()` 成功解析的比例 | 统计历史数据 / 回归测试 | ≥95% | P0 |
| 1.8 | **兜底逻辑有效性** | LLM 返回空/异常时，是否正确使用 DEFAULT_QUESTIONS | 注入异常输入，检查输出 | 100% | P0 |

### ROUND_STRATEGIES 参考

```python
# 一轮（tech_initial）：第1道为自我介绍，至少1道行为面试题
# 二轮（tech_deep）：不需要自我介绍，基于简历深挖
# 三轮（hr_comprehensive）：至少2道行为面试题，1道综合案例题
```

### 评测代码示例

```python
from deepeval import assert_test
from deepeval.test_case import LLMTestCase
from deepeval.metrics import GEval
from deepeval.metrics.g_eval import EvaluationSteps

jd_relevancy = GEval(
    name="JD Relevancy",
    criteria="评估面试问题是否与目标职位描述（JD）高度相关。高分：问题直接考察 JD 中提到的核心技能。低分：问题与 JD 无关或过于泛化。",
    evaluation_params=[EvaluationSteps.INPUT, EvaluationSteps.ACTUAL_OUTPUT],
    threshold=0.7,
)

def test_plan_jd_relevancy(mock_api_config):
    plan = generate_interview_plan_sync(resume, jd, max_questions=5, api_config=mock_api_config)
    plan_text = "\n".join([f"{q['id']}. [{q['type']}] {q['content']}" for q in plan])
    test_case = LLMTestCase(input=f"JD: {jd}\n简历: {resume}", actual_output=plan_text)
    assert_test(test_case, [jd_relevancy])
```

## 维度 2：面试对话质量（Interview Dialogue Quality）

**评测对象**：`node_responder()` 的输出和 `_classify_responder_action()` 的判断

**输入**：候选人回答 + 当前问题 + 下一问题
**输出**：面试官回复 + 路由判断（follow_up / next_question）

### 指标列表

| # | 指标 | 量化方式 | 评测方法 | 阈值 | 优先级 |
|---|------|---------|---------|------|--------|
| 2.1 | **追问合理性** | 追问是否针对回答的薄弱环节 | DeepEval `GEval`：「追问是否与候选人回答中的薄弱点相关」 | ≥0.7 | P1 |
| 2.2 | **追问次数控制** | `follow_up_count` ≤ `max_follow_ups` | 断言检查 | 100% | P0 |
| 2.3 | **过渡自然度** | 题目之间的过渡是否流畅 | DeepEval `GEval`：「面试官的过渡是否自然」 | ≥0.7 | P2 |
| 2.4 | **分类准确性** | `_classify_responder_action()` 对 follow_up/next_question 的判断准确率 | 标注数据集 + 准确率/Precision/Recall | ≥90% | P0 |
| 2.5 | **幻觉率** | 面试官是否编造不存在的信息 | DeepEval `HallucinationMetric` | ≥0.8 | P1 |
| 2.6 | **工具使用合理性** | ReAct Agent 的 tool_calls 参数完整性和调用时机 | 自定义：检查 tool_calls 日志 | 无异常调用 | P2 |

### _classify_responder_action 测试用例

```python
class TestClassifyResponderAction:
    """追问分类函数的边界测试"""

    def test_empty_response_defaults_to_next(self):
        assert _classify_responder_action("", "q1", "q2") == "next_question"

    def test_contains_next_question_text(self):
        resp = "很好，接下来请介绍一下你对微服务的理解"
        next_q = "请介绍一下你对微服务的理解"
        assert _classify_responder_action(resp, "当前问题", next_q) == "next_question"

    def test_contains_follow_up_marker(self):
        resp = "你刚才提到了性能优化，能具体说说用了哪些方法吗？"
        assert _classify_responder_action(resp, "当前问题", "下一题") == "follow_up"

    def test_question_mark_without_next(self):
        resp = "这个方案的并发处理是怎么做的？"
        assert _classify_responder_action(resp, "当前问题", "下一题") == "follow_up"

    def test_no_question_mark_no_marker(self):
        resp = "很好，你的回答很全面"
        assert _classify_responder_action(resp, "当前问题", "下一题") == "next_question"
```

## 维度 3：候选人评分准确性（Scoring Accuracy）

**评测对象**：`interview_analysis.process_interview_summary()` 的输出

**输入**：面试对话历史
**输出**：评分报告（综合评分 + 6 维度评分 + evidence + improvement_tip）

### 指标列表

| # | 指标 | 量化方式 | 评测方法 | 阈值 | 优先级 |
|---|------|---------|---------|------|--------|
| 3.1 | **评分校准度** | AI 评分 vs 人工标注评分的 MAE | 构建标注数据集（≥30 对），计算统计指标 | MAE ≤15 分 | P1 |
| 3.2 | **Pearson 相关系数** | AI 评分与人工评分的线性相关 | 统计计算 | ≥0.7 | P1 |
| 3.3 | **证据支撑度** | `evidence` 字段是否充分支撑分数 | DeepEval `GEval`：「证据是否充分支撑评分」 | ≥0.7 | P1 |
| 3.4 | **建议可操作性** | `improvement_tip` 是否具体、可执行 | DeepEval `GEval`：「建议是否具体可操作」 | ≥0.7 | P2 |
| 3.5 | **跨维度一致性** | 6 个维度的分数标准差（非全部中庸） | 统计分析：标准差 ≥10 | 标准差 ≥10 | P2 |
| 3.6 | **报告结构完整性** | 评分报告是否包含所有必要字段 | 断言检查：overall_score, dimensions, summary 等 | 100% | P0 |

### 评分维度参考

```python
# 6 个评分维度（来自 unified_report.py）：
# - technical_skill: 技术能力
# - communication: 沟通能力
# - problem_solving: 问题解决
# - learning_ability: 学习能力
# - team_collaboration: 团队协作
# - overall_potential: 综合潜力
```

## 维度 4：简历优化质量（Resume Optimization Quality）

**评测对象**：`resume_optimizer_graph.optimize_resume()` 的输出

**输入**：原始简历 + JD + 面试对话（可选）
**输出**：优化结果（match_score + optimized_sections + change_items + keyword_analysis）

### 指标列表

| # | 指标 | 量化方式 | 评测方法 | 阈值 | 优先级 |
|---|------|---------|---------|------|--------|
| 4.1 | **JD 关键词覆盖** | 优化后内容覆盖 JD 关键词的比例 | 自定义：关键词提取 + 匹配 | ≥70% | P0 |
| 4.2 | **事实忠实度** | 优化内容是否偏离原始简历事实 | DeepEval `FaithfulnessMetric` | ≥0.8 | P0 |
| 4.3 | **变更合理性** | `change_items` 的 `change_type` 分布 | 统计：`fact_inference` 类型占比不应过高 | fact_inference ≤30% | P1 |
| 4.4 | **反思有效性** | `reflect` 节点是否真正改进了方案 | 自定义：对比 moderator_summary 与 refined_result 的差异 | 有实质改进 | P2 |
| 4.5 | **HR 反馈一致性** | HR 审核官的 pass_rate 与 match_score 方向一致 | 自定义：两者差异 ≤20 | 差异 ≤20 | P2 |
| 4.6 | **报告结构完整性** | 输出是否包含所有必要字段 | 断言检查 | 100% | P0 |

### 评测代码示例

```python
from deepeval.metrics import FaithfulnessMetric

def test_resume_faithfulness(mock_api_config):
    """简历优化不能编造原始简历中没有的事实"""
    original = "负责公司内部 OA 系统的后端开发，使用 Python + Django"
    optimized = optimize_resume_sync(original, jd, api_config=mock_api_config)

    faithfulness = FaithfulnessMetric(threshold=0.8)
    test_case = LLMTestCase(input=original, actual_output=str(optimized))
    assert_test(test_case, [faithfulness])
```

## 维度 5：Agent 工作流正确性（Workflow Correctness）

**评测对象**：LangGraph 状态机的路由逻辑和节点执行

**特点**：纯 pytest 单元测试，**不消耗任何 LLM tokens**

### 指标列表

| # | 指标 | 量化方式 | 评测方法 | 阈值 | 优先级 |
|---|------|---------|---------|------|--------|
| 5.1 | **路由正确性** | `route_entry()` 给定 state 返回正确路由 | 参数化测试：多组 state → 期望路由 | 100% | P0 |
| 5.2 | **Responder 后路由正确性** | `route_after_responder()` 的判断 | 参数化测试：idx vs plan 长度 | 100% | P0 |
| 5.3 | **追问分类边界** | `_classify_responder_action()` 的边界条件 | 参数化测试：空文本、含问号、含关键词等 | 100% | P0 |
| 5.4 | **图结构正确性** | 节点和边的数量与名称 | 检查 graph.nodes 和 graph.edges | 与设计一致 | P1 |
| 5.5 | **兜底逻辑** | DEFAULT_QUESTIONS 在异常时正确返回 | 注入空 plan / 异常 API | 返回默认问题 | P0 |
| 5.6 | **简历优化图流转** | 8 个节点的执行顺序 | Mock LLM，验证节点调用序列 | prepare→3专家→moderator→reflect→refine→finalize | P1 |

### 评测代码示例

```python
import pytest
from app.services.interview.interview_graph import route_entry, route_after_responder
from langgraph.graph import END

class TestRouteEntry:
    @pytest.mark.parametrize("plan,expected", [
        ([], "planner"),                    # 无计划 → planner
        ([{"content": "q1"}], "responder"), # 有计划 → responder
    ])
    def test_route_entry(self, plan, expected):
        state = build_test_state(interview_plan=plan)
        assert route_entry(state) == expected

class TestRouteAfterResponder:
    def test_all_done_goes_to_summary(self):
        state = build_test_state(current_question_index=5, interview_plan=[{}] * 5)
        assert route_after_responder(state) == "summary"

    def test_has_more_ends(self):
        state = build_test_state(current_question_index=2, interview_plan=[{}] * 5)
        assert route_after_responder(state) == END
```
