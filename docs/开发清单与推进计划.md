# AI 面试助手开发清单与推进计划

## 1. 文档目的

本文件是 `V1初版方案与功能现状.md` 的执行版，重点回答下面几件事：

- 先做什么，后做什么
- 每个功能改哪些后端文件、前端文件、数据表
- 哪些是新增能力，哪些是优化现有能力
- 每个阶段做到什么程度算完成

建议原则：

- 改动尽量小，优先复用现有 `resume`、`session`、`analysis`、`profile` 体系
- 先补数据模型，再补接口，再补前端交互
- 每做完一个模块，都补最基本的接口测试和手工回归清单

---

## 2. 总体推进顺序

建议分 4 个阶段推进。

### 阶段 0：准备与收口

目标：

- 把现有代码的“可复用基础能力”梳理清楚
- 统一文档、接口返回结构、前端状态口径

本阶段任务：

- [ ] 更新现有文档，统一当前功能口径
- [ ] 统一 `resume` 相关接口返回字段命名
- [ ] 统一报告结构字段命名
- [ ] 盘点当前数据库表和缺失表
- [ ] 盘点前端状态中已存在和缺失的类型
- [ ] 明确 PostgreSQL、Redis、题库、RAG、鉴权边界

完成标准：

- 文档和真实代码口径一致
- 后续开发不再频繁返工接口命名

### 阶段 1：先补最值钱的求职前链路

目标：

- 让“简历 + JD”链路先闭环

本阶段模块：

1. JD 匹配分析
2. 候选人素材库与 JD 自动组装简历
3. 项目经历重写助手

### 阶段 2：补面试复盘和求职闭环

目标：

- 让“面试后还能继续用”

本阶段模块：

1. 面试后短板地图
2. 投递追踪与复盘

### 阶段 3：体验和质量增强

目标：

- 把已有能力从“能用”提升到“可持续使用”

本阶段模块：

1. 评分解释性
2. 多轮面试上下文继承
3. 面试题题库与 Agentic RAG 检索
4. 报告结构标准化
5. 简历优化可控性

## 9. 阶段 3：优化现有能力

### 9.1 面试评分解释性

目标：

- 把“结论”变成“有证据的结论”

建议改动：

- 在 `analysis_service.py` 中新增证据提取
- 在 `interview_analysis.py` 中补引用片段
- 前端报告中展示“回答片段 -> 问题 -> 扣分原因 -> 改进建议”

建议字段：

- `evidence`
- `reason`
- `better_answer_example`
- `improvement_tip`

### 9.2 多轮面试上下文继承

目标：

- 下一轮面试自动追问上一轮薄弱项

建议改动：

- `session_services/interview_plan.py`
- `core/interview_planner.py`
- `core/graph.py`

建议做法：

- 从上一轮 `weakness_report` 读取短板
- 作为下一轮面试计划生成输入
- 在计划中标记 `follow_up_priority`

### 9.3 面试题题库与 Agentic RAG 检索

目标：

- 让题目不再只是一次性生成结果，而是沉淀为可检索、可复用的长期资产

建议改动：

- 新增题库模块，支持用户上传题库
- 允许用户把系统生成的面试题确认后纳入题库
- Agentic RAG 继续归类为阶段 3 的旧链路增强，不单独前移成前置大功能
- `core/interview_planner.py` 在阶段 3 接入 `interview_rag.py` 或同类检索层
- 题目生成要能引用 `jd_analysis_results`、`candidate_materials`、`interview_weakness_reports`、`question_bank_items`
- 第一版先用 PostgreSQL 结构化检索和全文检索验证价值，语义召回不足时再优先补 `pgvector`

建议字段：

- `source_type`
- `source_id`
- `origin_session_id`
- `retrieval_score`

## 15. 近期实际执行建议

如果你现在准备开始动手，建议第一批直接做这 3 件事：

1. 先补 `JD 匹配分析`
- 价值最直接
- 风险最低
- 和现有 `resume` 体系最容易融合

2. 再补 `candidate_materials + resume_assembly_results`
- 后面项目改写、自动组装、投递记录都要依赖它

3. 然后做 `项目经历重写助手`
- 用户感知最强
- 最容易形成差异化

完成这 3 项后，再做短板地图和投递追踪，整体节奏会顺很多。
