# Mem0 长短期记忆系统引入计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 以 mem0 作为项目长期记忆主系统，让 AI 面试智能体跨会话记住用户偏好、候选人事实、历史短板、复练目标和投递策略，并与当前 FastAPI + LangGraph 面试流程稳定集成。

**Architecture:** 短期记忆继续由当前 PostgreSQL `sessions/messages` 与 LangGraph `MemorySaver` 承担；长期记忆统一交给 mem0 管理，应用层只负责检索、注入、写入触发、审计展示和用户隔离。第一版采用应用代码主动 `search → prompt 注入 → LLM 响应 → 后台 add` 的模式，不让模型自由决定是否调用记忆工具，保证行为稳定。

**Tech Stack:** FastAPI、LangGraph、LangChain Core、mem0ai、pgvector、OpenAI-compatible LLM/Embedding、PostgreSQL、SQLAlchemy。

---

## 1. 说明

- **mem0 是长期记忆主系统**。
- PostgreSQL 不再自建 `agent_memories` 作为主事实表。
- PostgreSQL 只保留当前会话、消息、画像、短板、题库等业务真值。
- mem0 负责长期记忆的提取、语义检索、更新、冲突处理和历史追踪。
- pgvector 作为 mem0 默认向量存储，复用现有 PostgreSQL 实例。
- 现有 Agentic RAG 继续处理题库、JD、简历素材、短板报告等业务检索，不和 mem0 混成一个索引。

推荐边界：

```text
短期会话记忆：PostgreSQL messages + LangGraph state
长期用户记忆：mem0
业务事实检索：现有 PostgreSQL / 后续 Agentic RAG
向量记忆存储：pgvector（复用现有 PostgreSQL），由 mem0 管理
```

## 2. 当前项目接入点

### 2.1 已有基础

- `backend/app/api/chat.py`：文本面试主入口，适合接入记忆检索与写入。
- `backend/app/api/voice_chat.py`：语音面试入口，需要复用同一套记忆服务。
- `backend/app/services/interview/interview_graph.py`：LangGraph 面试图，适合在 `InterviewState` 中增加 `memory_context`。
- `backend/app/services/interview/interview_planner.py`：面试计划生成，可使用 mem0 召回的长期偏好、短板和练习目标。
- `backend/app/services/interview/interview_analysis.py`：面试总结和短板生成后，适合把稳定短板写入 mem0。
- `backend/app/repositories/session/repo_impl/message_mgmt.py`：当前 user / assistant 消息保存点，可作为记忆写入触发依据。
- `backend/app/services/memory.py`：当前是 LangGraph checkpointer 文件，避免新增同名 package；mem0 服务建议放到 `backend/app/services/agent_memory/`。

### 2.2 需要解决的问题

- 当前没有长期记忆层。
- `user_id` 在部分链路中仍使用 `default_user`，接入 mem0 前必须统一用户隔离。
- 当前面试图只拿当前会话状态，不能主动召回跨会话偏好和历史短板。
- 当前 RAG 主要服务面试题生成，不负责用户长期记忆。
- mem0 写入会调用 LLM 和 embedding，不能阻塞 SSE 主响应。

## 3. mem0 方案设计

### 3.1 总体链路

```text
用户发起面试 / 回答问题
  ↓
Chat API / Voice Chat API
  ↓
SessionRepo 读取当前会话状态
  ↓
AgentMemoryService.search_memories(user_id, query)
  ↓
把 Top-K 长期记忆格式化为 memory_context
  ↓
interview_graph planner / responder / summary 使用 memory_context
  ↓
LLM 生成问题、追问、总结
  ↓
保存 user / assistant 消息到 PostgreSQL
  ↓
后台 AgentMemoryService.add_interaction(...)
  ↓
mem0 自动提取、去重、更新、保存长期记忆
```

### 3.2 mem0 负责什么

| 能力 | 是否交给 mem0 | 说明 |
| --- | --- | --- |
| 长期事实提取 | 是 | 调用 `add()` 时由 mem0 判断哪些内容值得记 |
| 长期记忆语义检索 | 是 | 调用 `search()` 召回相关记忆 |
| 长期记忆更新与冲突处理 | 是 | mem0 根据已有记忆判断 ADD / UPDATE / DELETE / NONE |
| 记忆变更历史 | 是 | 使用 `history(memory_id=...)` 查询 |
| 全部记忆查看 | 是 | 使用 `get_all(user_id=...)` |
| 记忆删除 | 是 | 使用 `delete(memory_id=...)` / `delete_all(user_id=...)` |
| 当前会话完整消息 | 否 | 继续放 `messages` 表 |
| 面试计划、短板报告、题库 | 否 | 继续放业务表和 Agentic RAG |
| 严格事务型状态 | 否 | 继续由 PostgreSQL 管理 |

### 3.3 记忆分类

mem0 支持 `metadata`，项目侧应统一写入 metadata，便于检索、展示和删除。

推荐类型：

| `memory_type` | 示例 | 来源 |
| --- | --- | --- |
| `preference` | 用户希望反馈先给结论，再给细节 | 用户明确表达 |
| `candidate_fact` | 候选人有 FastAPI + PostgreSQL 项目经验 | 简历、用户回答、用户确认 |
| `weakness` | 系统设计回答缺少容量估算 | 面试总结、短板报告 |
| `practice_goal` | 下一轮重点练 Redis 缓存一致性 | 用户目标、复盘建议 |
| `delivery_strategy` | 某岗位主推支付项目经历 | JD 匹配、投递复盘 |
| `safety_preference` | 不保存手机号等敏感信息 | 用户隐私偏好 |

metadata 示例：

```python
metadata = {
    "memory_type": "weakness",
    "source": "interview_summary",
    "session_id": session_id,
    "round_index": round_index,
    "round_type": round_type,
    "confidence": 0.86,
    "project": "agent_interview",
}
```

### 3.4 用户隔离策略

mem0 至少使用 `user_id` 隔离。建议固定：

```text
user_id = 当前请求 X-User-ID 或登录用户 ID
agent_id = "agent_interview_agent"
run_id = session_id
```

使用规则：

- 用户长期偏好、候选人事实、短板：按 `user_id` 写入。
- 当前单次会话临时上下文：可附加 `run_id=session_id`，但不把完整聊天流水作为长期记忆保存。
- 不同智能体未来拆分时，用 `agent_id` 区分，如 `interview_agent`、`resume_agent`。

### 4.3 环境变量

新增到 `env_example`、`.env.production.example`：

```env
# mem0
MEM0_ENABLED=true
MEM0_VECTOR_PROVIDER=pgvector
MEM0_PGVECTOR_COLLECTION=mem0_memories
MEM0_PGVECTOR_HOST=localhost
MEM0_PGVECTOR_PORT=5432
MEM0_PGVECTOR_DBNAME=postgres
MEM0_PGVECTOR_USER=postgres
MEM0_PGVECTOR_PASSWORD=

# mem0 LLM，用于记忆提取和冲突判断
MEM0_LLM_PROVIDER=openai
MEM0_LLM_MODEL=deepseek-chat
MEM0_LLM_API_KEY=${DEEPSEEK_API_KEY}
MEM0_LLM_BASE_URL=https://api.deepseek.com/v1

# mem0 embedding，用于语义检索
MEM0_EMBEDDER_PROVIDER=openai
MEM0_EMBEDDER_MODEL=text-embedding-3-small
MEM0_EMBEDDER_API_KEY=${OPENAI_API_KEY}
```

注意：

- embedding 维度必须和模型一致。若换成其他 embedding 模型，必须同步调整 `MEM0_EMBEDDING_DIMS`。
- `MEM0_PGVECTOR_*` 配置可复用现有 PostgreSQL 连接信息，与业务数据库共用同一实例。

### 6.2 AgentMemoryService

文件：`backend/app/services/agent_memory/service.py`

业务接口：

```python
class AgentMemoryService:
    async def search_memories(
        self,
        *,
        user_id: str,
        query: str,
        limit: int = 5,
        memory_types: list[str] | None = None,
    ) -> list[dict]:
        ...

    async def add_interaction(
        self,
        *,
        user_id: str,
        session_id: str,
        user_message: str,
        assistant_message: str,
        metadata: dict,
    ) -> dict:
        ...

    async def add_summary_memory(
        self,
        *,
        user_id: str,
        session_id: str,
        content: str,
        memory_type: str,
        metadata: dict,
    ) -> dict:
        ...

    async def get_all(self, *, user_id: str, page_size: int = 100) -> list[dict]:
        ...

    async def history(self, *, user_id: str, memory_id: str) -> list[dict]:
        ...

    async def delete(self, *, user_id: str, memory_id: str) -> None:
        ...
```

实现原则：

- 对外隐藏 mem0 返回结构差异，统一转成项目 DTO。
- `search_memories()` 兼容 mem0 返回 `{"results": [...]}` 或分页结构。
- `add_interaction()` 默认后台执行。
- `history()` 和 `delete()` 前先校验 memory 是否属于当前 `user_id`。
- `MEM0_ENABLED=false` 时返回空结果，并跳过写入。

### 6.3 写入过滤

文件：`backend/app/services/agent_memory/filters.py`

mem0 会自行判断是否值得保存，但项目侧仍需做第一层节流：

跳过写入：

- 用户消息少于 8 个中文字符或 20 个英文字符。
- 消息只包含“好的”“谢谢”“继续”“可以”“嗯”等低价值内容。
- assistant 回复为空或异常中断。
- 明显包含 API Key、身份证、手机号等敏感信息，除非后续有明确的隐私授权策略。

允许写入：

- 用户明确说“记住”“以后都”“我偏好”“我的项目是”。
- 面试回答中包含候选人稳定项目事实。
- 面试总结中包含稳定短板和复练目标。
- 用户对简历、投递策略做明确确认。

### 6.4 Prompt 格式化

文件：`backend/app/services/agent_memory/formatter.py`

格式：

```text
[长期记忆]
以下内容来自 mem0 长期记忆，可能随时间变化。若与用户当前输入冲突，以当前输入为准，并先确认是否更新记忆。

1. [preference] 用户希望面试反馈先给结论，再给原因。
2. [weakness] 用户最近系统设计回答中缺少容量估算。
3. [practice_goal] 下一轮重点练习 Redis 缓存一致性。
```

限制：

- 默认最多 5 条。
- 总字符数不超过 `MEM0_CONTEXT_CHAR_LIMIT`。
- 不把 memory 当 system 指令，只作为用户历史事实参考。

## 7. 接入 LangGraph 面试流程

### 7.1 修改 InterviewState

文件：`backend/app/services/interview/interview_graph.py`

新增字段：

```python
memory_context: str
memory_items: list[dict]
```

用途：

- `memory_context` 给 planner / responder / summary prompt 使用。
- `memory_items` 用于日志、调试和后续 SSE 事件展示。

### 7.2 面试开始时检索

文件：`backend/app/api/chat.py` 的 `start_interview()`

检索 query：

```text
JD + 公司信息 + 简历摘要 + 面试轮次 + round_type
```

召回类型：

- `preference`
- `candidate_fact`
- `weakness`
- `practice_goal`
- `delivery_strategy`

目的：

- planner 根据历史短板规划本轮题目。
- responder 按用户偏好调整表达方式。
- 不用重复问用户已经长期确认过的信息。

### 7.3 流式回答时检索

文件：`backend/app/api/chat.py` 的 `stream_chat()`

检索 query：

```text
当前用户回答 + 当前问题 + 下一题主题 + 当前轮次
```

目的：

- 判断是否需要根据历史短板追问。
- 让过渡语结合用户长期目标。
- 在总结时避免重复建议。

### 7.4 后台写入

文件：`backend/app/api/chat.py` 的 `event_generator()`

时机：

```text
保存 user 消息
保存 assistant 消息
发送 done 事件
后台 AgentMemoryService.add_interaction()
```

原则：

- `search()` 需要在响应前执行，因为结果要进 prompt。
- `add()` 放后台，因为它影响未来轮次，不影响当前回复。
- 写入失败只记录日志，不让本轮聊天失败。

## 8. 记忆管理 API

文件：`backend/app/api/memory.py`

接口：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/memory` | 查看当前用户全部 mem0 记忆 |
| `GET` | `/api/memory/search?q=` | 搜索当前用户记忆 |
| `GET` | `/api/memory/{memory_id}/history` | 查看单条记忆变更历史 |
| `DELETE` | `/api/memory/{memory_id}` | 删除当前用户某条记忆 |
| `DELETE` | `/api/memory` | 清空当前用户全部记忆，需显式确认参数 |

安全规则：

- 所有接口必须使用 `X-User-ID` 或登录用户 ID。
- `delete_all` 必须要求 `confirm=true`，避免误删。
- 删除单条记忆前先 `get_all(user_id)` 校验归属。
- API 不暴露其他用户的 memory id、metadata 和 history。

## 9. 与现有 Agentic RAG 的关系

mem0 与 `docs/开发拆分计划/07-阶段3-Agentic-RAG使用场景与向量存储方案.md` 的边界如下：

| 场景 | 使用 mem0 | 使用 Agentic RAG |
| --- | --- | --- |
| 用户表达偏好 | 是 | 否 |
| 跨会话短板和练习目标 | 是 | 可作为业务报告检索补充 |
| JD、题库、简历素材检索 | 否 | 是 |
| 面试题生成证据来源 | mem0 提供用户历史背景 | RAG 提供业务证据 |
| 题目去重 | 可参考历史记忆 | 主要依赖 messages / question_bank / RAG |
| 记忆审计 | mem0 history | 业务表审计 |

推荐注入顺序：

```text
System role
  ↓
面试官规则
  ↓
mem0 长期记忆上下文
  ↓
Agentic RAG 证据上下文
  ↓
当前会话消息 / 当前用户回答
```

约束：

- mem0 记忆不能替代 RAG 证据。
- RAG 文档和题库内容不能写入 mem0，除非用户明确确认为个人长期偏好或事实。
- mem0 记忆与当前用户输入冲突时，以当前输入为准，并可在响应后触发更新。

## 12. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| mem0 SDK 返回结构随版本变化 | `AgentMemoryService` 做适配层，API 层不直接调用 mem0 |
| pgvector embedding 维度不一致 | 用 `MEM0_EMBEDDING_DIMS` 集中配置，换模型时重建 collection |
| `add()` 阻塞响应 | 写入后台执行，失败只记录日志 |
| 记忆误写 | 项目侧先过滤低价值和敏感内容，管理 API 支持删除 |
| 跨用户泄露 | 所有 `search/get_all/delete` 强制带 `user_id` |
| mem0 不可用 | `MEM0_ENABLED=false` 或异常降级为空记忆上下文 |
| 记忆与当前输入冲突 | Prompt 明确“以当前输入为准，并先确认是否更新记忆” |

## 13. 最小可行版本范围

第一版只做：

1. mem0 + pgvector 初始化。
2. `AgentMemoryService` 适配层。
3. 文本面试 `start_interview()` / `stream_chat()` 的检索注入。
4. 文本面试 `event_generator()` 的后台写入。
5. 基础记忆管理 API。
6. 单元测试与 mock 集成测试。

第一版不做：

- 自建 PostgreSQL 长期记忆主表。
- 把题库、JD、简历素材写入 mem0。
- 让 LLM 自主调用 `save_memory` / `search_memory` 工具。
- 前端复杂记忆管理页面。
- mem0 graph 功能。

这样能先把 mem0 长期记忆闭环跑通，再逐步扩展到语音面试、复盘、画像和投递策略。
