# 求职者端配置、启动与功能验收指南

本文只覆盖求职者端，用于第一次运行项目和启动后的手工验收。项目没有 HR 工作台；前端中的面试、简历、题库、投递和能力画像都服务于求职者。

更完整的环境变量说明见 [配置说明.md](./配置说明.md)。本文以“先跑通，再按需开启增强能力”为主。

## 1. 配置边界

项目有两类配置，作用不同：

| 配置 | 放置位置 | 说明 |
|---|---|---|
| 数据库、Redis、任务队列、RAG、mem0、Langfuse、文件上传、语音运行默认值 | 项目根目录 `.env` | 服务端统一运行配置，不提交到 Git |
| 求职者自己的模型提供商、API Key、Base URL、模型名及 Smart/Fast/Voice 等通道分配 | 前端左下角“设置” | 持久化在当前浏览器本地；调用模型时会随请求发给本项目后端，不写入 `.env` 或数据库 |

语音配置也遵守这个边界：`.env` 负责声音、输入输出格式和后端缺省模型；“设置”页的 Voice 通道负责实际使用的模型连接和求职者 API Key。

> 不要把真实 API Key、数据库密码、Fernet 密钥、Langfuse Secret 或 BOSS 浏览器目录提交到仓库、截图或日志中。下文均使用占位值。

## 2. 环境要求

- Python `>=3.11,<3.13`
- Node.js `>=20`
- Docker 与 Docker Compose
- `uv`
- PostgreSQL 16 + pgvector（本文通过 Docker 启动）
- Redis 7（仅异步首题队列需要）
- BOSS 半自动化另需在宿主机安装 Playwright Chromium；首次操作时在项目专用浏览器中登录 BOSS

## 3. 准备根目录 `.env`

以下命令均在项目根目录执行：

```bash
cp env_example .env
```

### 3.1 推荐的首次本地配置

先在 `.env` 中确认或覆盖以下配置。首次验收先关闭可选外部依赖，能减少排障变量：

```bash
POSTGRES_USER=agent_interview
POSTGRES_PASSWORD=replace_with_a_local_password
POSTGRES_DB=agent_interview
DATABASE_URL=postgresql://agent_interview:replace_with_a_local_password@localhost:5432/agent_interview

NEXT_PUBLIC_API_URL=http://localhost:8000

TASK_QUEUE_ENABLED=false
MEM0_ENABLED=false
RAG_VECTOR_ENABLED=false
RAG_AGENTIC_MODE=off
LANGFUSE_ENABLED=false

VOICE_MODEL=qwen3-omni-flash-2025-12-01
VOICE_NAME=Cherry
VOICE_INPUT_FORMAT=wav
VOICE_OUTPUT_FORMAT=wav
VOICE_TRANSCRIPT_TERM_FIXES={}
```

`POSTGRES_PASSWORD` 与 `DATABASE_URL` 中的密码必须一致。首次创建数据库卷后再修改密码，不会自动修改数据库中的账号密码。

`VOICE_TRANSCRIPT_TERM_FIXES` 是可选的 JSON 术语纠错表，只处理浏览器生成的实时转写文本。默认 `{}` 不改写候选人原话；确认某些术语长期被识别错误时再配置，例如：

```bash
VOICE_TRANSCRIPT_TERM_FIXES={"lang graph":"LangGraph","拉姆达":"Lambda"}
```

修改后重启后端生效。纠错后的文本会进入会话记录和后续复盘，原始音频仍保存在当前浏览器本地。

### 3.2 开启异步首题队列（可选）

队列模式需要 Redis、Worker 和 Fernet 密钥。生成密钥：

```bash
uv run --project backend python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

把输出写入 `.env`，不要粘贴到文档或提交记录：

```bash
TASK_QUEUE_ENABLED=true
REDIS_URL=redis://localhost:6379/0
TASK_PAYLOAD_ENCRYPTION_KEY=replace_with_generated_fernet_key
```

API 与 Worker 必须读取同一个 `TASK_PAYLOAD_ENCRYPTION_KEY`。

### 3.3 开启向量检索（可选）

当前向量服务实际读取以下变量，而不是 `env_example` 中预留的 `EMBEDDING_API_*`：

```bash
RAG_VECTOR_ENABLED=true
OPENAI_API_KEY=replace_with_embedding_api_key
OPENAI_BASE_URL=https://your-openai-compatible-endpoint/v1
EMBEDDING_MODEL=text-embedding-3-small
EMBEDDING_DIM=1536

RAG_AGENTIC_MODE=shadow
RAG_AGENTIC_MAX_ROUNDS=2
RAG_AGENTIC_MAX_QUERIES=4
RAG_AGENTIC_MIN_SCORE=0.30
RAG_AGENTIC_MIN_EVIDENCES=2
RAG_AGENTIC_MIN_SOURCE_TYPES=2
```

建议先用 `shadow` 观察结果与延迟，确认正常后再切到 `active`。不配置 Embedding 时保持 `RAG_VECTOR_ENABLED=false`，结构化题库和面经检索仍可使用。

### 3.4 开启 mem0 或 Langfuse（可选）

这两项不是首次启动必需能力。需要时按 [配置说明.md](./配置说明.md) 填写完整的 mem0 LLM/Embedding 或 Langfuse 凭据，再把对应开关设为 `true`。凭据不完整时保持关闭。

## 4. 本地开发启动（推荐）

这种方式支持前后端热更新，PostgreSQL/Redis 在 Docker 中运行。

### 4.1 启动基础设施

首次推荐配置关闭了任务队列，因此只需 PostgreSQL：

```bash
docker compose --env-file .env up -d postgres
```

如果 `TASK_QUEUE_ENABLED=true`，同时启动 Redis：

```bash
docker compose --env-file .env up -d postgres redis
```

### 4.2 安装并启动后端

```bash
cd backend
uv sync
```

数据库分两种情况：

- 全新数据库：先启动一次后端，让应用创建完整表结构；日志显示表结构已同步后停止后端，执行 `uv run alembic stamp head`，再重新启动。
- 已有数据库：启动新代码前执行 `uv run alembic upgrade head`。

启动后端：

```bash
uv run uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

全新数据库完成首次建表后，在另一个终端执行：

```bash
cd backend
uv run alembic stamp head
```

然后重启后端。当前首个 Alembic revision 是既有 schema 的增量变更，空库不要直接执行 `upgrade head`。

如果开启了任务队列，再开一个终端启动 Worker：

```bash
cd backend
uv run dramatiq app.services.agent_runs.worker --processes 1 --threads 1
```

### 4.3 安装并启动前端

再开一个终端：

```bash
cd web
npm install
npm run dev
```

当前 `web/next.config.ts` 没有加载父目录 `.env`。标准本地开发不受影响，因为前端代码的缺省 API 地址就是 `http://localhost:8000`；如果后端使用自定义地址，请在启动前端时显式传入：

```bash
NEXT_PUBLIC_API_URL=http://your-backend-address npm run dev
```

全容器模式会从根目录 `.env` 把该值作为前端构建参数传入，不需要在 `web/` 下再复制一份配置。

访问地址：

- 前端：<http://localhost:3000>
- 后端健康检查：<http://localhost:8000/health>
- 后端 API 文档：<http://localhost:8000/docs>

可先验证后端：

```bash
curl http://localhost:8000/health
```

预期返回包含 `"status":"healthy"` 的 JSON。

## 5. 全容器启动（可选）

全容器模式会启动 PostgreSQL、Redis、后端、Worker、前端和 Nginx。`docker-compose.yml` 当前会为后端和 Worker 开启任务队列，因此必须配置 Fernet 密钥。

`.env` 至少确认：

```bash
POSTGRES_USER=agent_interview
POSTGRES_PASSWORD=replace_with_a_strong_password
POSTGRES_DB=agent_interview
NEXT_PUBLIC_API_URL=http://localhost
TASK_QUEUE_ENABLED=true
TASK_PAYLOAD_ENCRYPTION_KEY=replace_with_generated_fernet_key
```

全新数据库：

```bash
docker compose --env-file .env up -d --build
docker compose exec backend alembic stamp head
docker compose ps
```

已有数据库升级：

```bash
docker compose --env-file .env build backend worker frontend
docker compose --env-file .env up -d postgres redis
docker compose --env-file .env run --rm --no-deps backend alembic upgrade head
docker compose --env-file .env up -d
```

访问 <http://localhost>，健康检查为：

```bash
curl http://localhost/health
```

`NEXT_PUBLIC_API_URL` 是前端构建参数，修改后需要重新构建 `frontend`。

## 6. 首次进入：配置求职者模型

建议先完成此步骤，再验收业务功能。

1. 打开前端，点击左侧底部“设置”。
2. 在“添加模型”区域点击 `+`。
3. 选择提供商，填写 API Key、Base URL 和模型；提供商可能自动填入 Base URL，以实际供应商文档为准。
4. 点击“测试连接”。
5. 看到“连接成功！”后点击“保存”。未测试通过时不能保存。
6. 至少为面试功能分配 `Smart` 和 `Fast` 两个通道。首次验收可以让二者指向同一个已验证模型。
7. 要验收完整简历工作流，再分配“通用任务、匹配分析师、内容优化师、HR 审核官、质量审核”；首次可复用同一个模型，后续再按成本与质量拆分。
8. 要使用语音面试，新增模型名精确为 `qwen3-omni-flash-2025-12-01` 的连接，并在“语音模型”中将它分配给 Voice 通道。
9. 点击“完成”。

预期效果：

- 测试成功后出现“连接成功”状态，模型显示在已配置列表和各通道下拉框中。
- 通道选择实时保存到当前浏览器本地，刷新页面仍保留。
- 未配置 Smart/Fast 时，开始面试会提示先配置 API。
- 未配置 Voice 时，“语音对话”卡片不可选，并显示“请先在设置中配置语音模型”。

## 7. 前端功能验收清单

建议按下面顺序验收。涉及模型的结果受供应商、模型能力和网络影响，文档只约定页面和业务状态，不约定具体生成文案。

### 7.1 文字模拟面试

操作：

1. 左侧切到“模拟面试”，点击“新建模拟面试”。
2. 点击“上传简历”，推荐使用后端默认允许的 PDF、DOCX 或 TXT 文件。
3. 点击“目标岗位”区域，粘贴 JD 后点击“确认”；“公司信息”为选填。
4. 把“面试问题数量”设为 3–10；可设置“从个人题库抽取”的数量。
5. 选择“文字对话”，点击“开始面试”。
6. 在输入框回答并发送；需要帮助时点击提示按钮。
7. 完成本轮后点击“本轮能力画像”或“短板地图”；未到第 3 轮时还可点击“开启下一轮”。

预期：

- 上传成功后显示文件名和“已上传”，页面的“本次面试上下文”显示简历、目标岗位和个性化题目状态。
- 页面先显示准备/排队状态，然后流式出现首题；题库题不足时由模型补足。
- 发送回答后出现下一题并更新进度；提示按钮展示“回答提示”。
- 面试结束后出现本轮报告入口；同一岗位最多可继续到 3 轮，最后一轮不再显示下一轮入口。
- 左侧会话列表和“题库 → 面试历史”中能看到已保存的面试记录。

### 7.2 语音模拟面试

前置条件：Voice 通道已配置为 `qwen3-omni-flash-2025-12-01`，浏览器允许麦克风权限，系统有可用的输入/输出设备。

操作：

1. 按文字面试相同方式上传简历、填写 JD、设置题量。
2. 选择“语音对话”，点击“开始语音面试”。
3. 第一次进入时允许浏览器使用麦克风。
4. 面试官播报完问题后直接回答；说完点击“结束回答”。
5. 可用“静音 / 取消静音”控制麦克风，点击“结束面试”可提前结束。
6. 至少完成一题后退出语音界面，再从左侧会话列表重新进入该会话。

预期：

- 页面显示“AI 语音面试官”、题目进度和“正在连接 / 面试官正在听 / 你正在说话 / 面试官正在思考”等状态。
- 面试官问题以音频播放并同步显示字幕；回答时页面显示实时转写和音量动效。
- 点击“结束回答”后进入思考状态，随后播放下一题；题号随之更新。
- 后端会整理实时转写中的多余空白；配置 `VOICE_TRANSCRIPT_TERM_FIXES` 后，会按纠错表修正常见技术术语，再写入会话记录。
- 语音过程的音频保存在当前浏览器本地存储中，不应在页面刷新后意外上传为公开文件。
- 重新进入已有语音会话时恢复历史消息和题目进度，不重复创建开场白；可以从当前问题继续回答。
- 麦克风被拒绝、Voice 模型不兼容或供应商不支持所需音频格式时，应出现明确错误，不应无期限停在连接状态。

### 7.3 单轮复盘、能力画像与短板地图

操作：

1. 完成一轮面试后点击“本轮能力画像”。
2. 在弹窗中切到“短板地图”，点击“生成短板地图”。
3. 左侧“模拟面试”模式下点击“综合能力画像”，首次点击“生成能力画像”。

预期：

- 单轮能力画像显示总评、优势、待改进项和各维度评分；评分详情可包含证据、原因、改进建议和更好回答示例。
- 短板地图显示短板分类、典型问题分析、按优先级排列的改进行动和推荐练习题；可“重新生成”。
- 综合能力画像聚合已完成的面试历史；没有完成面试时应保持空状态或提示先完成面试。

> 评分、画像和改进建议仅用于求职者个人练习与复盘，不代表真实招聘结果，也不替代人工判断。

### 7.4 简历工具

左侧切到“简历工具”，主页面包含“竞争力分析、JD 匹配、内容优化”。左侧历史区分为“分析记录、JD 匹配、我的简历”。

#### 竞争力分析

1. 输入/导入简历内容，可补充目标 JD。
2. 点击“开始竞争力分析”。

预期：显示竞争力分析结果、维度评分、优势和可执行建议；完成结果进入“分析记录”。

#### JD 匹配

1. 输入简历和目标 JD。
2. 点击“开始 JD 匹配分析”。

预期：显示总体匹配度、技能/项目/经验/教育等维度、命中关键词和差距；结果进入“JD 匹配”。

#### 内容优化与生成简历

1. 输入简历和目标 JD，点击“生成内容优化建议”。
2. 查看 P1/P2 优先级建议，点击“生成优化简历”。
3. 如弹窗要求补充信息，填写后开始生成。

预期：先流式显示各分析节点进度，再给出具体修改建议；生成后的完整简历进入“我的简历”，可再次打开预览和编辑。

### 7.5 我的题库、文件导题和面经采集

#### 我的题库

1. 点击左侧“题库”进入“我的题库”。
2. 点击“添加题目”，填写题目；参考答案可选，并选择难度、题型。
3. 点击“保存”。

预期：提示“添加成功”，新题出现在列表并可被搜索/筛选；删除时第一次点击只提示“再点一次确认删除”，第二次才删除。

#### 从文件导题

1. 在“我的题库”点击“选择文件”，上传 PDF 或 Markdown。
2. 等待解析，勾选题目后点击“导入所选 N 题”。

预期：先显示解析出的候选题，导入成功后题目进入“我的题库”。

#### 面经采集

1. 切到“面经采集”。
2. 输入关键词后点击“采集并抽题”；也可粘贴本人有权使用的导出 JSON。
3. 勾选题目，选择导入题库或带入面试配置。

预期：显示采集到的面经数和候选题数；导入题库后可在“我的题库”查到，带入面试后新建面试页会提示有面经题等待用于本次面试。

> 不要上传 Cookie，不要绕过平台签名或验证码，只处理本人有权使用的内容。

### 7.6 投递追踪

1. 点击左侧“投递追踪”。
2. 点击“新建投递”，填写必填的公司名称、岗位名称；投递渠道、优先级和备注可选。
3. 点击“创建”，再点击新卡片进入详情。
4. 在详情中修改资料、添加事件或“添加复盘笔记”，点击“保存”。

预期：

- 新记录出现在列表，默认可按“全部、已收藏、已投递、面试中、Offer、已拒绝、已接受”筛选。
- 卡片显示公司、岗位、当前状态、优先级、渠道和更新时间。
- 添加事件后，详情中的时间线和列表状态随最新事件更新；删除操作需要确认。

### 7.7 BOSS 半自动化

前置条件：主后端运行在 Docker，宿主机能显示桌面浏览器，Smart/Fast 模型可用。根目录 `.env` 已配置至少 32 字符的 `BOSS_AUTOMATION_SERVICE_TOKEN`；`BOSS_BROWSER_PROFILE_DIR` 留空时默认使用 `backend/data/browser_profiles/boss`，不要提交、同步该目录，也不要指向日常 Chrome 默认 profile。

先在宿主机启动浏览器服务，再启动 Compose：

```bash
cd backend
uv run playwright install chromium
uv run uvicorn boss_service:app --host 0.0.0.0 --port 8765
```

```bash
docker compose --env-file .env up -d --build
```

1. 点击左侧“BOSS 半自动化”，停留在“批量抓取”。
2. 填写搜索关键词、可选城市、前 N 个（1–10）和简历内容。
3. 点击“一键批量抓取”。
4. 首次使用先在项目打开的专用浏览器中登录；如果出现滑块验证码，也在该窗口手动完成，页面最长等待约 3 分钟。
5. 查看“本次抓取结果”，再切到“已采集岗位”。

预期：

- Docker 后端通过鉴权 HTTP RPC 调用宿主机 Playwright；采集、预览和发送共享同一登录 profile。
- 前 N 个岗位生成 JD 分析、定制简历和 3 条打招呼文案，成功时提示抓取数量。
- 结果保存到“已采集岗位”，刷新后仍可查看或删除。
- 当前求职者端页面负责采集和生成投递资产，不应预期它未经人工确认自动投递或自动发送消息。

## 8. 建议的最小验收顺序

为了快速判断问题在哪一层，建议按以下顺序：

1. `curl /health` 成功。
2. 前端能打开，刷新无网络错误。
3. “设置”中添加一个模型并“测试连接”成功。
4. Smart/Fast 分配完成，文字面试能出现首题并完成一轮。
5. 简历“竞争力分析”能返回结果。
6. 题库能新增一题，并在新面试中抽取。
7. 投递追踪能新建记录并添加事件。
8. 配好 Qwen3-Omni 后再验收语音面试。
9. 最后按需开启队列、RAG、mem0、Langfuse 和 BOSS 半自动化。

## 9. 常见问题

### 后端提示数据库连接失败

检查 PostgreSQL 是否运行，以及 `.env` 中 `POSTGRES_PASSWORD` 与 `DATABASE_URL` 是否一致：

```bash
docker compose ps postgres
```

### 前端请求仍发到旧地址

本地 Next.js 不会自动读取父目录 `.env`，标准本地地址无需配置；自定义地址请用 `NEXT_PUBLIC_API_URL=http://your-backend-address npm run dev` 重新启动。全容器模式修改根目录 `.env` 后必须重新构建前端：

```bash
docker compose --env-file .env build frontend
docker compose --env-file .env up -d frontend nginx
```

### 开始面试时返回 503 或一直排队

若开启队列，检查 Redis、Worker 和 Fernet 密钥：

```bash
docker compose ps redis
```

本地只想先跑通时，把 `.env` 中 `TASK_QUEUE_ENABLED=false`，重启后端，不启动 Worker。

### 模型返回 401 / 403

在“设置”中重新打开对应模型，核对 API Key、Base URL 和模型名，再点“测试连接”。不要把 API Key 复制到日志或问题截图中。

### “语音对话”不可选

必须先新增模型名精确为 `qwen3-omni-flash-2025-12-01` 的配置，并把它分配到 Voice 通道。之后刷新新建面试页，并确认浏览器麦克风权限已允许。

### 简历上传格式不一致

前端文件选择器还会显示 DOC、MD，但后端默认 `ALLOWED_FILE_EXTENSIONS=pdf,docx,txt`。首次验收优先使用 PDF、DOCX 或 TXT；如确需其他格式，先统一修改 `.env` 允许列表并确认后端解析器支持。

### 查看服务日志

本地开发直接查看后端、Worker 和前端各自终端。全容器模式使用：

```bash
docker compose logs -f backend worker frontend
```

日志中不应打印 API Key、Fernet 密钥或浏览器凭据。
