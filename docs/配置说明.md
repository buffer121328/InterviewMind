### 1.1 首次启动必须修改

至少修改以下配置：

- `POSTGRES_PASSWORD`，并同步修改 `DATABASE_URL`；
- `TASK_PAYLOAD_ENCRYPTION_KEY`；
- `NEXT_PUBLIC_API_URL`；
- 如果启用向量检索，填写 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`EMBEDDING_MODEL`；
- 如果启用 mem0，填写 `MEM0_LLM_API_KEY` 和 `MEM0_EMBEDDER_API_KEY`；
- 在前端配置至少一个 Smart 模型和一个 Fast 模型。

生成任务加密密钥：

```bash
uv run --project backend python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

如果只想先启动基础面试功能，可以暂时使用：

```dotenv
TASK_QUEUE_ENABLED=false
RAG_VECTOR_ENABLED=false
MEM0_ENABLED=false
LANGFUSE_ENABLED=false
```

### 1.2 配置归属

| 配置类型 | 配置位置 | 是否包含密钥 |
|---|---|---|
| PostgreSQL、Redis、队列、RAG、mem0、Langfuse | 根目录 `.env` | 可能包含 |
| Smart/Fast 默认模型 | 前端「设置 → 模型配置」 | 是 |
| Fast Pool / Reasoning Pool | 前端「设置 → 模型配置」 | 是 |
| Voice 与专家通道 | 前端「设置 → 模型配置」 | 是 |
| 模型池失败阈值和冷却时间 | 根目录 `.env` | 否 |

> 前端模型配置保存在当前浏览器本地，并随业务请求发送给后端。不要在公共电脑保存生产 API Key。

## 3. 本地开发启动

这种方式便于热更新和调试：PostgreSQL / Redis 在 Docker 中，后端、Worker 和前端在本机运行。

### 3.1 配置环境变量

```bash
# 复制示例配置
cp env_example .env

# 编辑 .env，至少设置数据库密码、前端 API 地址和队列密钥
```

建议追加或修改：

```bash
POSTGRES_USER=agent_interview
POSTGRES_PASSWORD=change_me
POSTGRES_DB=agent_interview
DATABASE_URL=postgresql://agent_interview:change_me@localhost:5432/agent_interview
NEXT_PUBLIC_API_URL=http://localhost:8000
```

如果保持 `TASK_QUEUE_ENABLED=true`，生成 Fernet 密钥：

```bash
uv run --project backend python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

将输出写入根目录 `.env` 的 `TASK_PAYLOAD_ENCRYPTION_KEY`。API 和 Worker 必须使用同一个值，不要将密钥提交到仓库。

### 3.2 启动 PostgreSQL 和 Redis（Docker）

```bash
# 在项目根目录执行
docker compose --env-file .env up -d postgres redis
```

这会在 `localhost:5432` 启动 PostgreSQL、在 `localhost:6379` 启动 Redis。必须在首次创建 PostgreSQL 数据卷前确定密码；修改 `.env` 不会自动修改已创建的数据库账号。

### 3.3 安装依赖 & 启动后端

```bash
cd backend

# 安装后端依赖
uv sync

# 已有数据库：启动新代码前执行增量迁移
uv run alembic upgrade head

# 启动后端服务（开发模式）
uv run uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

#### 全新数据库

当前 Alembic 的首个 revision 是针对既有 schema 的增量变更，不能在空库直接执行 `upgrade head`。首次启动顺序为：

1. 运行上述 `uvicorn` ，等日志出现表结构已同步后停止。
2. 在 `backend` 目录执行 `uv run alembic stamp head`。
3. 再次启动后端。

#### 启动队列 Worker

队列开启时，在新终端执行：

```bash
cd backend
uv run dramatiq app.services.agent_runs.worker --processes 1 --threads 1
```

本地不想运行 Redis / Worker 时，设置 `TASK_QUEUE_ENABLED=false`，首题将走同步兼容路径。

### 3.4 启动前端

```bash
cd web

# 安装前端依赖
npm install

# 启动开发服务器
npm run dev
```

访问：
- 后端 API 文档：http://localhost:8000/docs
- 前端界面：http://localhost:3000

进入前端后，还需在「设置 → 模型配置」中至少配置 `smart` 通道。主业务不会把 `.env` 中的服务端 Key 自动当成用户模型配置。

## 4. 环境变量详解

所有配置通过项目根目录的 `.env` 文件进行管理。

### 4.1 数据库配置

```bash
# PostgreSQL 数据库地址（完整 DSN）
DATABASE_URL=postgresql://agent_interview:change_me@localhost:5432/agent_interview

# 或者分别指定：
POSTGRES_USER=agent_interview       # 数据库用户名
POSTGRES_PASSWORD=change_me         # 数据库密码，请替换
POSTGRES_DB=agent_interview         # 数据库名称
POSTGRES_HOST=localhost             # 数据库主机
POSTGRES_PORT=5432                  # 数据库端口
```

> **优先级**：如果设置了 `DATABASE_URL`，则优先使用；否则从 `POSTGRES_*` 变量拼接。

### 4.2 RAG Embedding 配置

```bash
OPENAI_API_KEY=
OPENAI_BASE_URL=https://api.openai.com/v1
EMBEDDING_MODEL=text-embedding-3-small
EMBEDDING_DIM=1536
```

这是 `backend/app/services/rag/embedding_service.py` 当前实际读取的变量，需要 OpenAI 兼容的 Embedding API。若暂时不配置 Embedding，设置 `RAG_VECTOR_ENABLED=false`，保留结构化检索。

主业务 LLM 由前端「设置 → 模型配置」随请求传入。`FAST_LLM_*`、`SMART_LLM_*` 和 `DEEPSEEK_API_*` 当前不是主业务的服务端兜底。

### 4.3 前端配置

```bash
NEXT_PUBLIC_API_URL=http://localhost:8000   # 后端 API 地址
```

### 4.4 mem0 长期记忆配置

mem0 是项目的长期记忆系统，记住用户偏好和面试历史。

```bash
# 功能开关
MEM0_ENABLED=false                   # 配好 LLM 与 Embedding Key 后再启用

# 向量存储（复用 PostgreSQL）
MEM0_VECTOR_PROVIDER=pgvector
MEM0_PGVECTOR_COLLECTION=mem0_memories
MEM0_PGVECTOR_HOST=localhost
MEM0_PGVECTOR_PORT=5432
MEM0_PGVECTOR_DBNAME=agent_interview
MEM0_PGVECTOR_USER=agent_interview
MEM0_PGVECTOR_PASSWORD=change_me

# mem0 使用的 LLM（用于记忆提取）
MEM0_LLM_PROVIDER=openai
MEM0_LLM_MODEL=deepseek-chat
MEM0_LLM_API_KEY=
MEM0_LLM_BASE_URL=https://api.deepseek.com/v1

# mem0 Embedding（用于语义检索）
MEM0_EMBEDDER_PROVIDER=openai
MEM0_EMBEDDER_MODEL=text-embedding-3-small
MEM0_EMBEDDER_API_KEY=
MEM0_EMBEDDING_DIMS=1536

# 运行时参数
MEM0_SEARCH_LIMIT=5                  # 每次搜索返回的记忆条数
MEM0_CONTEXT_CHAR_LIMIT=1200         # 上下文最大字符数
MEM0_BACKGROUND_WRITE=true           # 是否后台写入记忆
```

> **关键配置**：`MEM0_LLM_MODEL` 和 `MEM0_LLM_API_KEY` 可以指向便宜的模型（如 deepseek-chat），用于记忆提取，不消耗主力模型的 tokens。

### 4.5 混合 Agentic RAG

面试检索保留低延迟的题库 / 面经快速路径，只在证据不足时运行有界 Agentic Retrieval，并可合并长期记忆。运行时爬虫不在面试热路径内，面经应先导入数据库。

```bash
RAG_VECTOR_ENABLED=true
RAG_AGENTIC_MODE=shadow
RAG_AGENTIC_MAX_ROUNDS=2
RAG_AGENTIC_MAX_QUERIES=4
RAG_AGENTIC_MIN_SCORE=0.30
RAG_AGENTIC_MIN_EVIDENCES=2
RAG_AGENTIC_MIN_SOURCE_TYPES=2
RAG_WEIGHT_VECTOR=0.45
RAG_WEIGHT_TEXT=0.25
RAG_WEIGHT_SOURCE=0.20
RAG_WEIGHT_FRESHNESS=0.10
```

| `RAG_AGENTIC_MODE` | 行为 | 建议用途 |
|---|---|---|
| `off` | 不运行 Agentic Retrieval | 故障隔离 / 最低延迟 |
| `shadow` | 运行并记录指标，不替换原结果 | 默认，先观察质量和延迟 |
| `active` | 只有证据质量改善时才采用 | 验证通过后开启 |

代码会将轮次限制在 1–2、查询数限制在 1–4。`rag_trace` 会记录是否触发 / 采用、轮次、候选数、来源分布和错误类型，便于评估后再从 `shadow` 切到 `active`。

### 4.6 统一可恢复任务中心与 LLM Fallback

以下任务默认统一走 Redis + Dramatiq：

- 面试首题生成；
- 简历优化；
- 面试完成后的本轮能力画像与短板报告；
- 岗位 JD 分析、定制简历与打招呼文案资产。

Redis 只保存 Broker 消息、模型池共享状态与互斥锁；简历、JD 和前端 API 配置以 Fernet 加密后存入 PostgreSQL，Worker 消息只包含任务 ID。任务中心提供分页列表、执行计划、等待任务立即取消、运行中协作取消、失败重试和陈旧任务主动恢复。

```bash
TASK_QUEUE_ENABLED=true
REDIS_URL=redis://localhost:6379/0
TASK_PAYLOAD_ENCRYPTION_KEY=  # API 与 Worker 共用，使用 Fernet.generate_key() 生成
LLM_REQUEST_TIMEOUT_SECONDS=45
LLM_MAX_TOKENS=8000
API_CONFIG_VALIDATION_TIMEOUT_SECONDS=10
LLM_POOL_FAILURE_THRESHOLD=2
LLM_POOL_COOLDOWN_SECONDS=60
AGENT_RUN_LOCK_TTL_SECONDS=600
AGENT_RUN_MAX_ATTEMPTS=3
AGENT_RUN_STALE_SECONDS=1800
AGENT_RUN_HEARTBEAT_SECONDS=30
AGENT_RUN_CANCEL_POLL_SECONDS=2
AGENT_RUN_RECOVERY_INTERVAL_SECONDS=60
RESUME_GENERATION_SESSION_TTL_HOURS=24
LLM_POOL_REDIS_ENABLED=true
LLM_POOL_INFLIGHT_TTL_SECONDS=600
ALLOW_PRIVATE_MODEL_BASE_URLS=true
```

`AGENT_RUN_HEARTBEAT_SECONDS` 控制 Worker 刷新任务状态的频率；Redis 运行锁也会按 TTL 的三分之一自动续租。`AGENT_RUN_CANCEL_POLL_SECONDS` 控制 Worker 检查运行中取消请求的频率。`AGENT_RUN_RECOVERY_INTERVAL_SECONDS` 控制 API 后台主动扫描陈旧任务的周期，不再依赖用户打开任务中心。`AGENT_RUN_STALE_SECONDS` 表示任务多久未被领取或没有心跳后可视为中断；陈旧的 `queued/retrying` 会重新投递，未超过 `AGENT_RUN_MAX_ATTEMPTS` 的陈旧 `running` 会恢复为 `retrying`。

交互式简历生成会话保存在 PostgreSQL，默认保留 `RESUME_GENERATION_SESSION_TTL_HOURS=24` 小时，不再依赖单个 API 进程的内存。

任务状态变化会持久化到 `agent_run_events`，并通过 `/api/agent-runs/{run_id}/events/stream` 提供带 sequence 和 `Last-Event-ID` 的可重放 SSE。运行中取消先进入 `cancel_requested`，Worker 取消当前异步执行后再进入 `cancelled`。

LangGraph checkpoint 使用异步 PostgreSQL saver；只有初始化失败时才回退到进程内 MemorySaver。

本地没有 Redis 时将 `TASK_QUEUE_ENABLED=false`，上述任务改走同步兼容路径。队列模式下 Broker 或加密密钥不可用会返回 `503`，任务会落为失败状态，可在修复配置后从任务中心重试。

### 4.7 文件解析配置

```bash
FILE_PARSE_TIMEOUT_SECONDS=30
```

当前文件服务只从环境变量读取解析超时。文件格式和体积限制由上传接口及服务代码校验，不使用旧的 `UPLOAD_DIR`、`MAX_FILE_SIZE_MB`、`ALLOWED_FILE_EXTENSIONS`、`MAX_RESUME_COUNT` 环境变量。

### 4.8 语音面试运行默认值

```bash
VOICE_MODEL=qwen3-omni-flash-2025-12-01
VOICE_NAME=Cherry
VOICE_INPUT_FORMAT=wav
VOICE_OUTPUT_FORMAT=wav
VOICE_TRANSCRIPT_TERM_FIXES={}
```

- 前四项是后端语音会话默认值；实际供应商、密钥和模型优先在前端“模型配置”的 `Voice` 通道中维护。
- `VOICE_TRANSCRIPT_TERM_FIXES` 是 JSON 格式的术语纠错表，默认不改写候选人原话。例如：

```bash
VOICE_TRANSCRIPT_TERM_FIXES={"lang graph":"LangGraph","拉姆达":"Lambda"}
```

修改后需重启后端。纠错后的文本会进入面试历史和复盘，原始录音仍只保存在浏览器本地。

### 4.9 BOSS 半自动化（可选）

```bash
BOSS_AUTOMATION_SERVICE_URL=http://host.docker.internal:8765
BOSS_AUTOMATION_SERVICE_TOKEN=
BOSS_AUTOMATION_REQUEST_TIMEOUT_SECONDS=240
BOSS_BROWSER_PROFILE_DIR=
BOSS_APPLY_APPROVAL_TTL_SECONDS=300
```

主后端和 Worker 可继续运行在 Docker 中；它们通过 `BOSS_AUTOMATION_SERVICE_URL` 调用宿主机 Playwright 服务。`BOSS_AUTOMATION_SERVICE_TOKEN` 必须在根目录 `.env` 中配置为至少 32 字符的随机值，宿主机服务和 Compose 共用该值，但不要提交或输出到日志。可执行 `openssl rand -hex 32` 生成。

采集、预览和发送统一使用宿主机的持久化 Playwright profile。`BOSS_BROWSER_PROFILE_DIR` 只由宿主机服务读取；留空时默认使用 `backend/data/browser_profiles/boss`。首次操作会打开项目专用浏览器，手动登录一次后即可复用状态。该目录包含登录凭据，不能提交、同步或多人共用，也不要指向日常 Chrome 的默认 profile。

首次启用前安装浏览器：

```bash
cd backend
uv run playwright install chromium
uv run uvicorn boss_service:app --host 0.0.0.0 --port 8765
```

宿主机服务启动后，再执行 `docker compose --env-file .env up -d --build`。Docker 容器不安装 Chromium，也不保存 BOSS 登录态。自动投递仍采用「预览截图 → 用户确认 → 一次性许可发送」；RPC 通信异常且无法确认是否已点击时会转为人工接管，避免重复发送。许可有效期范围为 30–1800 秒。

### 4.10 Langfuse Agent 可观测性（可选）

Langfuse 用于关联 Agent 根 span、LangChain 模型调用和工具调用。未启用或凭据不完整时自动降级，不影响面试与简历优化链路。

```bash
LANGFUSE_ENABLED=false
LANGFUSE_PUBLIC_KEY=
LANGFUSE_SECRET_KEY=
LANGFUSE_BASE_URL=https://cloud.langfuse.com
```

安装评测依赖并运行不调用评审模型的工具回归用例：

```bash
cd backend
uv sync --extra eval
uv run deepeval test run tests/eval/test_agent_tool_correctness.py -d failing
```

Langfuse 凭据只应保存在本地 `.env` 或部署平台的密钥管理中，不能提交到仓库。DeepEval 工具评测使用本地确定性适配器，不需要 `OPENAI_API_KEY`；已有带 `llm` 和 `eval` 标记的 GEval 评测需要单独配置评审模型，未配置时会跳过。

## 5. LLM 模型池与多通道配置

聊天模型配置不从服务器 `.env` 读取，而是由用户在前端「设置 → 模型配置」中维护。所有模型接口都需要兼容 OpenAI Chat Completions 格式；本地模型可通过 Ollama、vLLM、LM Studio 等提供的 OpenAI-compatible 地址接入。

### 5.1 默认通道与模型池

| 配置 | 用途 | 为空时的兼容行为 |
|---|---|---|
| **Smart** | 复杂任务的默认模型 | 必须配置 |
| **Fast** | 低延迟任务的默认模型 | 必须配置 |
| **Reasoning Pool** | 复杂分析、规划、总结的负载均衡池 | 使用 Smart 组成单成员池 |
| **Fast Pool** | 快速分类、问答、点评的负载均衡池 | 使用 Fast 组成单成员池 |
| **voice** | 实时语音面试 | 独立配置；客户端创建可回退到 Fast 配置 |
| **general** | 简历分析、主持和通用任务 | 未配置时使用后续 fallback |
| **match_analyst** | JD 匹配分析 | 未配置时使用后续 fallback |
| **content_writer** | 简历润色和内容生成 | 未配置时使用后续 fallback |
| **hr_reviewer** | HR 审核 | 未配置时使用后续 fallback |
| **reflector** | 质量复核 | 未配置时使用后续 fallback |

推荐结构：

```text
Fast Pool
├── DeepSeek Chat / Flash
├── GLM Flash
└── 本地小模型

Reasoning Pool
├── DeepSeek Reasoner
├── GLM 推理模型
└── 其他企业 OpenAI-compatible 接口
```

### 5.2 前端配置步骤

1. 打开「设置 → 模型配置」；
2. 点击 `+`，分别保存模型的名称、API Key、Base URL 和 Model；
3. 在 Smart 和 Fast 下拉框中选择兼容默认模型；
4. 在 `Reasoning Pool` 多选区勾选复杂任务候选；
5. 在 `Fast Pool` 多选区勾选低延迟候选；
6. 按需配置 Voice 和简历工具专家通道；
7. 使用模型配置中的连接验证确认 API Key、地址和模型名称可用。

池为空不是错误：后端会自动用 Smart 或 Fast 默认模型构造单成员池，因此旧浏览器配置无需迁移即可继续使用。

### 5.3 调度算法

同一模型池采用请求级加权轮询。前端当前发送的成员权重均为 `1`，因此默认是等权轮询；后端协议支持 `1–100` 的 `weight`。Redis 可用时，API 与 Worker 共享轮询游标、失败次数、冷却键和 in-flight 计数，并优先从当前并发数最低的成员中按权重选择。

```text
请求 1 → Fast A
请求 2 → Fast B
请求 3 → 本地模型
请求 4 → Fast A
```

后端会按 `base_url + model + API Key 指纹` 去重，不会把同一个配置重复加入候选链。默认 `LLM_POOL_REDIS_ENABLED=true`；Redis 暂时不可用时自动降级为进程内调度，并每隔约 30 秒尝试恢复连接，不阻断模型调用。

### 5.4 失败冷却和 Fallback

模型池健康策略由 `.env` 控制：

```dotenv
LLM_POOL_FAILURE_THRESHOLD=2
LLM_POOL_COOLDOWN_SECONDS=60
LLM_POOL_REDIS_ENABLED=true
LLM_POOL_INFLIGHT_TTL_SECONDS=600
ALLOW_PRIVATE_MODEL_BASE_URLS=true
LLM_REQUEST_TIMEOUT_SECONDS=45
```

- 成员连续失败达到阈值后进入冷却；
- 多实例共享成员 in-flight 计数，优先选择并发更少的候选；
- 冷却期间优先由同池其他成员承接请求；
- 如果全部成员都在冷却，允许重新探测，避免永久不可用；
- 结构化调用的第一个候选按调用参数有限重试，备用候选各尝试一次。
- 所有模型 Base URL 会执行出站地址校验；`ALLOW_PRIVATE_MODEL_BASE_URLS=true` 用于个人本地 Ollama/vLLM/LM Studio。若以后部署到共享网络，应设为 `false`。

请求候选顺序：

```text
smart 请求：Reasoning Pool → general → Fast Pool
fast 请求：Fast Pool → general → Reasoning Pool
专家请求：专家通道 → general → Reasoning Pool → Fast Pool
```

候选会去重，所有候选均失败后才返回最后一次模型错误。

Voice 客户端配置采用 `voice → fast`；但音频能力不能由普通文本模型自动补齐，因此生产环境应显式配置支持所需音频输入输出的 Voice 模型。

### 5.5 配置数据与安全

前端发送给后端的模型池成员格式如下，通常无需手写：

```json
{
  "smart": {
    "api_key": "***",
    "base_url": "https://api.example.com/v1",
    "model": "reasoning-default"
  },
  "fast": {
    "api_key": "***",
    "base_url": "https://api.example.com/v1",
    "model": "flash-default"
  },
  "reasoning_pool": [
    {
      "name": "DeepSeek Reasoner",
      "api_key": "***",
      "base_url": "https://api.deepseek.com/v1",
      "model": "deepseek-reasoner",
      "weight": 1
    }
  ],
  "fast_pool": [
    {
      "name": "GLM Flash",
      "api_key": "***",
      "base_url": "https://open.bigmodel.cn/api/paas/v4",
      "model": "glm-4.7-flash",
      "weight": 1
    }
  ]
}
```

模型配置保存在浏览器本地，并随业务请求发送给后端。队列模式下包含模型配置的任务载荷会使用 `TASK_PAYLOAD_ENCRYPTION_KEY` 加密后存入 PostgreSQL；Dramatiq 消息只传任务 ID。不要把真实 API Key 写进文档、日志或提交到 Git。

## 7. 全容器启动

`docker-compose.yml` 定义 PostgreSQL、Redis、后端 API、Worker、前端与 Nginx。PostgreSQL 和 Redis 仅绑定到本机回环地址，既可供本地开发后端使用，也不会直接暴露到局域网。

先在根目录准备 `.env`：

```bash
cp env_example .env
```

至少确认以下项：

```bash
POSTGRES_USER=agent_interview
POSTGRES_PASSWORD=change_me
POSTGRES_DB=agent_interview
NEXT_PUBLIC_API_URL=http://localhost
TASK_QUEUE_ENABLED=true
TASK_PAYLOAD_ENCRYPTION_KEY=replace_with_generated_fernet_key
```

`NEXT_PUBLIC_API_URL` 是前端构建参数；修改后需重新构建 `frontend`。Compose 会覆盖后端的数据库、Redis 和 mem0 主机名，不要手动改成容器 IP。

### 7.1 全新数据库

```bash
docker compose --env-file .env up -d --build
docker compose exec backend alembic stamp head
docker compose ps
```

后端首次启动会创建完整 schema，然后用 `stamp head` 标记当前 revision。

### 7.2 已有数据库升级

先构建镜像并只启动依赖，在新后端启动前执行迁移：

```bash
docker compose --env-file .env build backend worker frontend
docker compose --env-file .env up -d postgres redis
docker compose --env-file .env run --rm --no-deps backend alembic upgrade head
docker compose --env-file .env up -d
```

### 7.3 运维命令

```bash
# 查看状态与日志
docker compose ps
docker compose logs -f backend worker

# 停止
docker compose down
```

> **注意**：服务端口与本地已安装的 PostgreSQL 可能冲突。修改 `ports` 映射（如 `5433:5432`）然后更新 `.env` 中的端口即可。

## 8. 启动验证与常见问题

### 8.1 验证清单

```bash
# 本地开发
curl http://localhost:8000/health

# 全容器（通过 Nginx）
curl http://localhost/health
```

预期返回 `"status":"healthy"`。然后确认：

- 本地开发可访问 `http://localhost:3000` 和 `http://localhost:8000/docs`。
- 全容器可访问 `http://localhost`。
- `docker compose ps` 中 `postgres`、`redis`、`backend` 处于正常状态，`worker` 没有循环重启。

### 8.2 常见问题

### Q: 启动后端时报错 `connection refused`？

A: 确保 PostgreSQL 已启动。检查 `docker compose ps` 或本地 PostgreSQL 服务状态。

### Q: LLM 调用返回 401 / Unauthorized？

A: 检查 API Key 是否正确、是否过期。可以在前端「设置 → 模型配置」中点击「验证」按钮测试连通性。

### Q: mem0 记忆功能不工作？

A: 检查 `MEM0_ENABLED=true`，并确认 mem0 的 LLM 和 Embedding 配置正确。可以临时设置 `MEM0_ENABLED=false` 来绕过问题。

### Q: 数据库表不存在？

A: 先判断是全新库还是已有库。全新库使用 `create_all` 后 `stamp head`；已有库在启动新代码前执行 `alembic upgrade head`。不要在空库直接执行当前首个增量 revision。

### Q: 如何只启用部分通道？

A: 前端至少需要选择一个 Smart 默认模型和一个 Fast 默认模型。Reasoning Pool、Fast Pool、Voice 和专家通道都可以留空：空池会分别使用 Smart/Fast 构造单成员池，专家通道按 `general → Reasoning Pool → Fast Pool` 回退。

### Q: 多个 API/Worker 是否共享模型池状态？

A: 默认共享。确保所有实例使用同一个 `REDIS_URL`，并保持 `LLM_POOL_REDIS_ENABLED=true`。轮询游标、失败冷却和 in-flight 计数会写入 Redis；Redis 故障时各实例临时降级为进程内状态，恢复后自动重新接入共享调度。

### Q: Agentic RAG 延迟增加或检索异常？

A: 先将 `RAG_AGENTIC_MODE=off` 隔离 Agentic 路径。如果未配置 Embedding，同时设置 `RAG_VECTOR_ENABLED=false`。恢复时先用 `shadow` 观察 `rag_trace`，再切换到 `active`。

### Q: 如何安全管理配置？

A: 不要提交 `.env`、BOSS 浏览器 profile、API Key、Fernet 密钥或 Langfuse Secret Key。生产环境应使用部署平台的密钥管理，并替换所有示例密码。
