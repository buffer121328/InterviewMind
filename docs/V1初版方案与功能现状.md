# AI 面试助手 V1 初版方案与功能现状

## 1. 文档目的

本文档用于统一整理 `agent_interview` 当前项目的三类信息：

- 当前代码里已经实现的功能
- 当前项目仍待完善、待补强、待新增的功能点
- 结合 `Job` 与 `ResumeScreening` 两个项目思路后，面向求职/面试场景的 V1 初版方案

定位上，建议将本项目从“AI 面试工具”升级为“求职作战台”，核心链路为：

`简历上传 -> JD 分析 -> 简历优化/生成 -> 模拟面试 -> 报告复盘 -> 持续求职记录`

---

## 2. 当前项目已实现功能

## 3. 当前项目待完善的地方

下面这些并不一定是“完全没有”，有些是已经有基础能力，但产品化和结构化程度不够。

### 3.1 面试链路待完善

1. 面试评分解释性不够强
- 当前有画像和总结，但“为什么扣分、证据在哪、应该怎么答更好”还不够结构化

2. 多轮面试上下文继承仍可加强
- 已有轮次数据结构
- 但下一轮“自动针对上一轮短板追问”的能力还可以继续做强

3. 面后复盘还不够可执行
- 当前有总结和画像
- 但缺“薄弱点地图”“优先学习项”“推荐练习题”

4. 语音面试的稳定性还有工程化空间
- 断线重连
- 异常恢复
- 浏览器兼容性
- 长时会话资源控制

### 3.2 简历链路待完善

1. 简历优化结果可控性不足
- 需要明确区分：
  - 原文润色
  - AI 建议新增
  - 事实推断内容

2. 简历优化粒度还不够细
- 现在偏整份简历分析/整份优化
- 缺少“项目经历单段重写”“按模块单独改写”

3. 简历生成前的数据准备能力不完整
- 当前有 `generated_resumes`
- 但缺少“技术栈、项目、实习、教育、亮点素材先结构化沉淀，再按 JD 自动筛选组装”的产品能力

4. JD 匹配分析还不够独立
- 已有分析和优化能力
- 但缺一个单独、可复用、可量化的 JD 匹配评分模块

### 3.3 求职闭环待完善

1. 缺少投递追踪
- 没有记录岗位投递状态、轮次时间、反馈、offer、拒绝原因

2. 缺少求职任务看板
- 无法管理多个岗位、多个方向、多份简历、多次面试

3. 缺少以求职结果为目标的策略建议
- 如冲刺岗、稳妥岗、保底岗建议
- 当前更偏单次工具调用，缺“连续求职辅导”

### 3.4 数据和工程待完善

1. 数据模型还偏轻量
- 适合当前功能
- 但要支撑“求职作战台”，需要补岗位、候选人素材库、投递、事件流水、短板报告等表

2. 测试覆盖明显不足
- 当前项目目录中没有形成完整的后端自动化测试体系
- 前端也缺关键流程回归清单

3. 文档与当前代码存在轻微脱节
- 旧文档更多描述早期方案
- 需要持续更新到当前真实功能状态

## 4. 可新增的核心功能方向

以 `agent_interview` 为蓝本，参考：

- `Job`：更强的简历优化、岗位技术栈路由、面试材料生成
- `ResumeScreening`：更强的结构化解析、匹配分析、数据建模

建议新增的高价值功能如下。

### 4.1 JD 定制求职包

输入岗位 JD，一次产出：

- 定制版简历
- 定制自我介绍
- 重点项目改写
- 高频追问题清单
- 该岗位的知识补强建议

### 4.2 JD 匹配度评分

将当前简历与岗位做结构化对比，输出：

- 总匹配度
- 技能匹配度
- 项目匹配度
- 年限匹配度
- 缺失关键词
- 优先改进点

### 4.3 项目经历重写助手

针对单个项目支持：

- STAR 重写
- 指标量化
- 面试追问预测
- 风险表达修正
- 面向某个 JD 的定向改写

### 4.4 面试后短板地图

面试结束后沉淀：

- 卡壳题列表
- 短板知识分类
- 回答弱点说明
- 推荐复习顺序
- 推荐练习题

### 4.5 投递追踪与复盘

记录求职进度：

- 已收藏
- 已投递
- 一面 / 二面 / 终面
- offer
- rejected

并支持：

- 面试时间
- 面试反馈
- 挂因复盘
- 下次改进动作

### 4.6 候选人素材库与 JD 自动组装简历

不建议让用户长期维护很多份完整简历，更合理的方式是先沉淀结构化素材，再按岗位自动生成对应版本。

建议让用户维护的不是“简历版本”，而是“素材资产”：

- 技术栈清单
- 项目经历
- 实习经历
- 工作经历
- 教育经历
- 证书/奖项
- 自我介绍亮点

生成逻辑：

- 用户上传原始简历，系统先做结构化拆分
- 用户补充或修正技术栈、项目、实习等素材
- 输入 JD 后，系统自动筛选最相关的素材片段
- 再把筛选后的素材交给简历优化/生成模块，产出岗位定制简历

这样更接近真实求职流程，也比手工维护多份完整简历更稳。

### 4.7 求职策略建议

根据用户背景和目标岗位给出：

- 冲刺岗
- 稳妥岗
- 保底岗
- 当前最该补什么
- 先投什么再投什么

## 5. V1 初版方案

### 5.1 V1 目标

将项目从“AI 面试助手”升级为“求职作战台”，先覆盖这条最小可用链路：

`上传简历 -> 输入 JD -> 生成定制结果 -> 模拟面试 -> 形成复盘 -> 跟踪岗位进度`

### 5.2 V1 范围

V1 先做 5 个模块：

1. `JD 匹配分析`
2. `候选人素材库与 JD 自动组装简历`
3. `项目经历重写助手`
4. `面试后短板地图`
5. `投递追踪与复盘`

原因：

- 能直接补强当前最关键的求职链路
- 与现有代码复用度高
- 商业化价值清晰

### 5.3 每个模块的目标

#### 5.3.1 JD 匹配分析

输入：

- 简历内容
- 岗位 JD

输出：

- 总匹配度
- 维度分数
- 缺失点
- 关键词覆盖
- 优先改进建议

可参考：

- `ResumeScreening` 的结构化解析与筛选逻辑
- `Job` 的技术栈路由思想

#### 5.3.2 候选人素材库与 JD 自动组装简历

目标：

- 让用户先维护可复用素材，而不是维护很多份完整简历
- 根据 JD 自动从素材库中挑选更相关的技术栈、项目、实习、经历片段
- 将筛选结果交给现有简历优化/生成链路，输出岗位定制简历

建议素材颗粒度：

- 技术栈
- 项目经历
- 实习经历
- 工作经历
- 教育经历
- 奖项证书
- 可量化成果

#### 5.3.3 项目经历重写助手

目标：

- 支持单段项目的定向改写
- 强化量化指标、STAR 表达、岗位适配度

#### 5.3.4 面试后短板地图

目标：

- 把当前总结/画像升级成更可执行的复盘结果

建议输出维度：

- 基础概念
- 项目表达
- 系统设计
- 八股原理
- 行为面试
- 沟通表达

#### 5.3.5 投递追踪与复盘

目标：

- 让用户在系统内持续记录投递和复盘
- 从“一次性工具”变成“持续陪跑工具”

## 6. V1 优先优化现有能力

在做新增功能的同时，建议优先补这 5 项体验/质量问题。

### 6.1 简历优化可控性

需要增加：

- 明确标记 AI 推断
- 区分“润色”“改写建议”“可能不真实的补全”
- 支持用户逐条采纳建议

### 6.2 面试评分可解释性

建议每个维度都给：

- 引用的回答片段
- 扣分原因
- 更优回答示例
- 改进建议

### 6.3 多轮面试上下文继承

建议增强：

- 延续上一轮弱项
- 下一轮自动重点追问
- 避免重复问
- 形成成长轨迹

### 6.4 报告结构标准化

统一输出：

- 总评
- 优势
- 风险点
- 核心短板
- 推荐练习
- 下一步行动

### 6.5 候选人画像沉淀

将历史数据沉淀为长期画像：

- 技能标签
- 岗位偏好
- 项目亮点
- 语言表达问题
- 历史成长趋势

## 7. V1 开发顺序建议

建议按以下顺序推进：

1. `JD 匹配分析`
2. `候选人素材库与 JD 自动组装简历`
3. `项目经历重写助手`
4. `面试后短板地图`
5. `投递追踪与复盘`
6. `回头补评分解释、多轮继承、报告结构`

原因：

- 前 3 项最容易提升用户感知价值
- 第 4、5 项让用户留存更高
- 最后再补深度体验优化，风险更低

---

## 8. V1 完成标准

达到以下标准，可认为 V1 具备初步可用性：

- 用户能上传原始简历并保存
- 用户能输入 JD 并获得匹配分析
- 用户能生成至少一个定制版简历
- 用户能对单个项目经历做定向重写
- 用户能完成一次模拟面试并拿到结构化短板报告
- 用户能记录某岗位的投递进度和复盘
- 新增核心接口有基础测试
- 前端主流程不需要频繁跳页即可完成
